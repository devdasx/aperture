# Send Compose Feature Matrix (doc-grounded, 2026-06-15)
Authoritative per-chain spec for the Send amount/compose screen + fee/UTXO data layer. Researched by aperture-chain-data agents against official RPC/protocol docs (live-verified). Every fact is doc-cited.
**Scope:** 26 chains. Fee model + exact fetch RPC, gas/compute/weight, UTXO + coin selection, OP_RETURN/memo/destination-tag/comment/data, per-network extras, and wallet-core tx-construction notes for the future sign step.

---

## G1 Bitcoin family

### `bitcoin`
- **Fee model:** Pure sat/vB (satoshis per virtual byte). Fee = feeRate(sat/vB) x vsize(vB). vsize = ceil(weight/4) per BIP141; witness data discounted 4x. NO EIP-1559, NO gas — it is weight-based fee on the wire but the wallet works in the simpler sat/vB abstraction (this is what every UTXO wallet does, and what wallet-core's byte_fee field consumes). Slow/normal/fast presets derived DIRECTLY from the provider's recommended-fees object: economyFee/hourFee -> slow, halfHourFee -> normal, fastestFee -> fast. Each is already a sat/vB integer. A custom-fee field lets the user override with their own sat/vB. Live-verified 2026-06-15: mempool.space returned {fastestFee:1,halfHourFee:1,hourFee:1,economyFee:1,minimumFee:1} (current near-empty mempool). minimumFee is the relay floor — clamp any computed/preset rate to >= minimumFee (Bitcoin Core minrelaytxfee default 1000 sat/kvB = 1 sat/vB) or the broadcast is rejected with 'min relay fee not met'. Bitcoin Core's own RPC equivalent is estimatesmartfee which returns BTC/kB (CONSERVATIVE/ECONOMICAL modes, conf_target 1-1008) — but Aperture's registered providers are Esplora, not a full node, so use the Esplora/mempool fee endpoints below, not estimatesmartfee.
- **Fee fetch RPC:** PRIMARY (matches the registered balance/history provider mempool.space, fallback blockstream.info — both Esplora): `GET https://mempool.space/api/v1/fees/recommended` -> JSON {fastestFee,halfHourFee,hourFee,economyFee,minimumFee} ALL in sat/vB (live-verified 2026-06-15). Blockstream's Esplora has no /v1/fees/recommended; its equivalent is `GET https://blockstream.info/api/fee-estimates` -> JSON map {confTarget(string): sat/vB(float)} e.g. {"1":1.136,"6":1.014,"144":1.014,"1008":0.752} (live-verified 2026-06-15) — derive fast=estimates["1"], normal=estimates["6"], slow=estimates["144"]. Full-node alternative (not used here, documented for completeness): JSON-RPC estimatesmartfee, request {"jsonrpc":"2.0","method":"estimatesmartfee","params":[conf_target,"CONSERVATIVE"],"id":1} -> {feerate (BTC/kB), blocks}; convert BTC/kB -> sat/vB by feerate*1e8/1000.
- **Gas / compute / weight:** No gas. The wallet estimates VIRTUAL SIZE (vB) to compute fee = rate x vsize. wallet-core computes vsize internally and sizes the fee for you (byte_fee x transaction_size); for the compose-screen estimate BEFORE signing use the standard size model: vsize ≈ overhead(10.5 vB) + inputs x perInputVbytes + outputs x perOutputVbytes. Per-input vB by type: P2WPKH(bech32) ≈ 68 vB, P2PKH(legacy) ≈ 148 vB, P2SH-P2WPKH ≈ 91 vB, P2TR ≈ 57.5 vB. Per-output vB: P2WPKH ≈ 31, P2PKH ≈ 34, P2TR ≈ 43, OP_RETURN ≈ 8.5 + dataBytes. Native-transfer default = 1 typical input + 2 outputs (recipient + change). Token transfer: N/A — Bitcoin has no native fungible tokens in Aperture's scope (BRC-20/Runes/ordinals are out of scope; if added later they are still BTC txs, fee still sat/vB x vsize, just larger). The actual input count is NOT fixed — it is the result of coin selection (see utxo), so the compose screen must re-estimate the fee after selecting inputs, or use wallet-core's TransactionPlan which returns the exact fee + selected utxos.
- **Units & decimals:** Smallest unit = satoshi. 1 BTC = 100,000,000 sat (decimals = 8, confirmed in wallet-core registry coinId 0). Fee rate unit = sat/vB. Fee total computed in sat (integer), displayed as BTC = feeSat / 1e8 and as fiat via the price layer. All money math in Decimal (Rule). Recommended-fee values arrive as sat/vB integers; multiply by estimated vsize to get sat, divide by 1e8 for BTC display. existing BitcoinFamilyTransactionAdapter already uses satsPerCoin = 1e8 Decimal for all four chains.
- **UTXO:** YES — UTXO model. Fetch the address UTXO set (matches registered provider): `GET https://mempool.space/api/address/{address}/utxo` -> JSON array [{txid, vout, value(sat), status:{confirmed,block_height,block_time}}] (live-verified 2026-06-15 on bc1q…: real UTXOs returned with value in sat). Blockstream identical path. To spend a UTXO wallet-core needs the locking script — derive it locally via BitcoinScript.lockScriptForAddress (no extra fetch needed for own address) OR Esplora exposes scriptpubkey via `GET /api/address/{address}` txs. COIN SELECTION: implement Branch-and-Bound (BnB, Bitcoin Core default) with largest-first / accumulative fallback; or pass the full UTXO list to wallet-core and let its planner select (TransactionPlan returns chosen utxos + fee + change). Prefer CONFIRMED utxos (status.confirmed==true) unless user opts into spending unconfirmed. DUST LIMIT: P2PKH 546 sat, P2WPKH/P2TR 294 sat (= 3 x dustRelayFee where dustRelayFee=3000 sat/kvB; P2PKH=182 vB x 3, P2WPKH=98 vB x 3) — never create a change/recipient output below the recipient script's dust limit; instead drop the change into fee. CHANGE: send remainder to a fresh change address (own wallet); if change < dust, add it to fee (wallet-core does this when disable_dust_filter is false). RBF: signal opt-in BIP125 by setting input nSequence <= 0xFFFFFFFD (wallet-core OutPoint.sequence); Bitcoin Core v28+ relays full-RBF by default so replacement works either way, but signaling lets explorers/wallets show 'replaceable'. SEND-ALL (max): set use_max_amount=true (wallet-core) -> sends balance minus fee, one output, no change.
- **Data / memo / tag:** OP_RETURN supported. Format: a zero-value output whose script is OP_RETURN <push data>. Standardness limit (pre-Core v30): 80 bytes of data (datacarriersize default 83 incl. opcodes), exactly ONE OP_RETURN output per tx. Bitcoin Core v30 (Oct 2025) raised default datacarriersize to ~100,000 bytes and allows multiple OP_RETURN outputs — but for broad relay compatibility Aperture should keep to 80 bytes / 1 output. wallet-core field: output_op_return (Data) + output_op_return_index (optional position). EXCHANGES: Bitcoin deposits do NOT use OP_RETURN/memo/destination-tag — each user gets a unique deposit ADDRESS; the compose screen needs NO memo field for BTC sends to exchanges. OP_RETURN is an optional advanced feature (data anchoring), never required. NO destination tag, NO memo concept on BTC.
- **Other features:** ADDRESS TYPES the compose/sign must handle for recipient AND change: P2PKH (1…), P2SH (3…), P2WPKH bech32 (bc1q…), P2TR bech32m (bc1p…). Validate the recipient address checksum + network before enabling Send. RESERVE/MIN-BALANCE: none (no account reserve like XRP). SEND-ALL: use_max_amount (subtract fee from amount). FEE-BUMPING: RBF (replace) + CPFP (child-pays-for-parent via spending own unconfirmed change) — RBF needs the original inputs re-signed at higher rate; surface 'Speed up' later. UNCONFIRMED-INPUT policy: allow toggling whether to spend unconfirmed change (own). MIN RELAY FEE clamp (1 sat/vB) on the slow preset. LOCKTIME: default 0 (wallet-core lock_time). The compose screen must collect: recipient address, amount (or max), fee preset/custom sat/vB, optional OP_RETURN data, optional RBF toggle. After input selection, recompute fee (vsize changes with input count). Live freshness: re-fetch UTXO set + fee rate immediately before signing (Rule #27 just-in-time sync) so nonce-equivalent (here: UTXO set + fee) is current.
- **Tx construction (wallet-core):** wallet-core (WalletCore) FULLY supports Bitcoin signing. SigningInput = BitcoinSigningInput. Required fields collected at compose time so signing works later: hashType (BitcoinScript.hashTypeForCoin(.bitcoin) = SIGHASH_ALL), amount (sat), byteFee (sat/vB from the preset), toAddress, changeAddress (own fresh change addr), utxo [BitcoinUnspentTransaction{outPoint{hash(little-endian txid), index(vout), sequence}, amount(sat), script=lockScriptForAddress}], privateKey[] (added at sign time, NOT compose). NATIVE MULTI-OUTPUT: yes — extra_outputs [OutputAddress{to_address, amount}] for batch sends beyond the primary to_address; output_op_return for data. use_max_amount for send-all. dust_policy.fixed_dust_threshold to set the dust cutoff (else default). plan: call AnySigner.plan(input,coin:.bitcoin) -> TransactionPlan {amount, available_amount, fee, change, utxos(selected), output_op_return} to PREVIEW exact fee + selected inputs on the compose screen before signing. coin_type not needed for BTC (only forks). Output of sign = BitcoinSigningOutput.encoded (raw tx hex) -> broadcast via POST /api/tx. COMPOSE MUST COLLECT NOW: the candidate UTXO set (with scripts), the chosen fee rate, recipient, amount/max, change address — these are exactly the SigningInput fields.
- **Doc URLs:** https://developer.bitcoin.org/reference/rpc/estimatesmartfee.html; https://mempool.space/docs/api/rest; https://mempool.space/api/v1/fees/recommended; https://github.com/Blockstream/esplora/blob/master/API.md; https://github.com/bitcoin/bitcoin/blob/master/src/policy/policy.cpp; https://en.bitcoin.it/wiki/OP_RETURN; https://github.com/glozow/bitcoin-notes/blob/master/datacarrier_28408.md; https://bitcoinops.org/en/topics/replace-by-fee/; https://bips.xyz/125; https://raw.githubusercontent.com/trustwallet/wallet-core/master/registry.json; https://raw.githubusercontent.com/trustwallet/wallet-core/master/src/proto/Bitcoin.proto; https://developer.trustwallet.com/developer/wallet-core/integration-guide/wallet-core-usage

### `bitcoinCash`
- **Fee model:** sat/vB, but BCH has NO segwit so size == vsize (every byte is a real byte; weight discount does not apply). Fee = feeRate(sat/byte) x txSize(bytes). Default min relay fee = 1 sat/byte (1000 sat/kB) — this is effectively the standard rate; BCH blocks are rarely full so 1 sat/byte confirms quickly. Slow/normal/fast: BCH has no fee market like BTC, so presets collapse near 1 sat/byte. Recommended source 1: BlockCypher /v1/bch/main {high/medium/low_fee_per_kb} (sat/kB -> /1000 = sat/byte). Recommended source 2 (simplest, matches registered Haskoin provider's ecosystem): default to 1 sat/byte normal, 1 sat/byte slow, 2 sat/byte fast — BCH wallets ship a near-fixed 1 sat/byte because the mempool is near-empty. Always clamp to >= 1 sat/byte (min relay).
- **Fee fetch RPC:** Registered provider Haskoin (api.haskoin.com) does NOT expose a fee-recommendation endpoint (UTXO/balance/tx only — confirmed). So fee fetch uses BlockCypher as the rate oracle: `GET https://api.blockcypher.com/v1/bch/main` -> {high_fee_per_kb, medium_fee_per_kb, low_fee_per_kb} in sat/kB; divide by 1000 for sat/byte (same field shape proven live for ltc/doce 2026-06-15). If BlockCypher unavailable, fall back to a fixed 1 sat/byte (the BCH network default min relay, reference.cash). Full-node equivalent: BCHN estimatefee RPC returns BCH/kB (legacy; BCH removed estimatesmartfee, uses fixed-fee estimatefee).
- **Gas / compute / weight:** No gas. Estimate raw byte size (NO witness discount on BCH): size ≈ 10 (overhead) + inputs x 148 (P2PKH input, BCH is P2PKH/P2SH only — no segwit) + outputs x 34 (P2PKH output). Native-transfer default = 1 input + 2 outputs ≈ 226 bytes -> ~226 sat at 1 sat/byte. Token transfer: SLP/CashTokens are out of Aperture's current scope; if added, still a BCH tx (fee = sat/byte x size, larger size for token script). wallet-core computes the exact size for you (byte_fee x transaction_size). Recompute after coin selection since size scales with input count.
- **Units & decimals:** Smallest unit = satoshi. 1 BCH = 100,000,000 sat (decimals = 8, wallet-core registry coinId 145). Fee rate unit = sat/byte (NOT sat/vB — no vbytes on BCH). Fee total in sat -> BCH = sat/1e8 -> fiat via price layer. Decimal math; existing adapter satsPerCoin = 1e8.
- **UTXO:** YES — UTXO model, P2PKH/P2SH only (no segwit, no taproot). Fetch UTXO set from the registered Haskoin provider: `GET https://api.haskoin.com/bch/address/{address}/unspent` -> JSON array [{address(cashaddr w/ bitcoincash: prefix), block:{height,position}, txid, index(vout), pkscript(hex locking script!), value(sat)}] (live-verified 2026-06-15 — returns pkscript inline, so NO extra script fetch needed; pass pkscript straight into wallet-core utxo.script). COIN SELECTION: same largest-first/BnB as BTC; or wallet-core planner. DUST LIMIT: 546 sat (P2PKH; reference.cash confirms 546-sat threshold) — provably-unspendable OP_RETURN outputs have dust threshold 0. CHANGE: fresh own change address; sub-dust change folds into fee. RBF: NOT supported/relevant on BCH — BCH removed RBF; transactions are first-seen-final, so do NOT signal RBF, set sequence = 0xFFFFFFFF (final). SEND-ALL: use_max_amount.
- **Data / memo / tag:** OP_RETURN supported. Format: OP_RETURN <push>. BCH limit AFTER HF-20210515: up to 220 bytes of data (script total 223 bytes = 1 opcode + 2 push + 220 data), and MULTIPLE OP_RETURN outputs allowed provided total data-output script size <= the limit (reference.cash). Larger than BTC's classic 80. wallet-core output_op_return. EXCHANGES: BCH deposits use a unique cashaddr ADDRESS — NO memo/destination-tag/OP_RETURN required. So the compose screen needs no mandatory memo field. OP_RETURN remains optional (memo.cash social, data). NO destination tag concept.
- **Other features:** ADDRESS FORMAT: CashAddr is the modern standard (q…/p… optionally prefixed bitcoincash:). Legacy 1…/3… (P2PKH/P2SH, same prefixes as BTC: 0/5) is still valid and wallet-core can derive it, but MOST EXCHANGES require CashAddr (some without the bitcoincash: prefix) — the compose screen should accept BOTH, normalize internally, and ideally display/validate as CashAddr. Aperture's Haskoin responses already carry the bitcoincash: prefix and the history adapter strips it for comparison (normalizedCashAddr) — reuse that. NO reserve/min-balance. NO RBF. SEND-ALL via use_max_amount. MIN RELAY 1 sat/byte clamp. Validate recipient is a valid BCH address (cashaddr checksum OR legacy base58check) and NOT mistakenly a BTC bech32 (bc1…) address — warn the user (common BCH/BTC confusion). Compose collects: recipient (cashaddr/legacy), amount/max, fee rate (sat/byte), optional OP_RETURN. Just-in-time re-fetch UTXO + rate before sign.
- **Tx construction (wallet-core):** wallet-core supports BCH natively (the docs' canonical signing example IS Bitcoin Cash). SigningInput = BitcoinSigningInput with: hashType = BitcoinScript.hashTypeForCoin(.bitcoinCash) — CRITICAL: BCH uses BIP143 sighash with the SIGHASH_FORKID flag (0x40) OR'd in (wallet-core sets this via hashTypeForCoin), distinct from BTC. coin_type = CoinType.bitcoinCash (145) — REQUIRED so wallet-core applies the fork rules. amount, byteFee (sat/byte), toAddress (cashaddr ok), changeAddress, utxo[BitcoinUnspentTransaction{outPoint{hash,index,sequence=0xFFFFFFFF}, amount, script = pkscript-from-Haskoin OR BitcoinScript.lockScriptForAddress(address:coin:.bitcoinCash)}], privateKey[] at sign time. NATIVE MULTI-OUTPUT: extra_outputs supported. use_max_amount for send-all. AnySigner.plan(...,coin:.bitcoinCash) for the fee/inputs preview. Output encoded -> broadcast. COMPOSE COLLECTS NOW: candidate UTXOs (with pkscript from Haskoin /unspent — already available), fee rate, recipient, amount/max, change addr.
- **Doc URLs:** https://developer.trustwallet.com/developer/wallet-core/integration-guide/wallet-core-usage; https://reference.cash/protocol/blockchain/transaction-validation/network-level-validation-rules; https://api.haskoin.com/; https://hackage.haskell.org/package/haskoin-store-0.64.7/changelog; https://www.blockcypher.com/dev/bitcoin/#blockchain-api; https://raw.githubusercontent.com/trustwallet/wallet-core/master/registry.json; https://raw.githubusercontent.com/trustwallet/wallet-core/master/src/proto/Bitcoin.proto; https://github.com/ActorForth/Auction-Protocol/blob/main/CHIP-2021-03-12_Multiple_OP_RETURN_for_Bitcoin_Cash.md

### `litecoin`
- **Fee model:** sat/vB (Litecoin HAS segwit + MWEB, so vsize/weight applies exactly like BTC; witness discounted 4x). Fee = feeRate(litoshi/vB) x vsize. Litecoin Core min relay fee default = 0.00001 LTC/kB (1000 litoshi/kvB = 1 litoshi/vB) BUT the wallet min tx fee floor is higher historically; in practice litecoinspace returns 1 sat/vB across all presets (near-empty mempool, live-verified 2026-06-15). Slow/normal/fast from litecoinspace recommended-fees: economyFee->slow, halfHourFee->normal, fastestFee->fast (identical Esplora shape to BTC). Custom litoshi/vB override allowed. Clamp to >= minimumFee returned by the endpoint (relay floor).
- **Fee fetch RPC:** Registered provider litecoinspace.org IS Esplora and exposes the SAME endpoint as mempool.space: `GET https://litecoinspace.org/api/v1/fees/recommended` -> {fastestFee,halfHourFee,hourFee,economyFee,minimumFee} in litoshi/vB (live-verified 2026-06-15: {fastestFee:1,...,minimumFee:1}). Fallback (registered fallback provider) BlockCypher: `GET https://api.blockcypher.com/v1/ltc/main` -> {high_fee_per_kb:12109, medium_fee_per_kb:8886, low_fee_per_kb:7271} in litoshi/kB (live-verified 2026-06-15) -> /1000 = litoshi/vB. Note BlockCypher's kB numbers reflect node-default richer fees (~8.9 sat/vB) vs litecoinspace's mempool-driven 1 sat/vB; prefer litecoinspace (mempool-accurate, matches the balance/history provider).
- **Gas / compute / weight:** No gas. vsize estimate identical to BTC (LTC has segwit): overhead 10.5 + inputs x perInputVbytes(P2WPKH 68 / P2PKH 148 / P2SH-P2WPKH 91) + outputs x perOutputVbytes(P2WPKH 31 / P2PKH 34 / OP_RETURN 8.5+data). Native-transfer default = 1 input + 2 outputs (recipient + change). LTC also has MWEB (MimbleWimble extension blocks) addresses (ltcmweb1…) — MWEB txs have a different size/fee model and wallet-core support is limited; treat MWEB as out-of-scope for v1 and only handle P2WPKH/P2PKH/P2SH. Token transfer: N/A (no native tokens). Recompute fee after coin selection.
- **Units & decimals:** Smallest unit = litoshi. 1 LTC = 100,000,000 litoshi (decimals = 8, wallet-core registry coinId 2). Fee rate = litoshi/vB. Fee total in litoshi -> LTC = /1e8 -> fiat. Decimal math; existing adapter satsPerCoin = 1e8 covers LTC.
- **UTXO:** YES — UTXO model with segwit. Fetch UTXO set from registered litecoinspace (Esplora): `GET https://litecoinspace.org/api/address/{address}/utxo` -> [{txid, vout, value(litoshi), status:{confirmed,block_height,block_time}}] (Esplora shape identical to BTC; note: must use a VALID LTC address — a BTC-style address returns 'Invalid Litecoin address', live-verified 2026-06-15). Locking script derived locally via BitcoinScript.lockScriptForAddress(coin:.litecoin) (own address) or from Esplora address txs scriptpubkey. COIN SELECTION: largest-first/BnB or wallet-core planner. DUST LIMIT: P2PKH 546 litoshi, P2WPKH 294 litoshi (same dustRelayFee 3000 formula as BTC). Historically Litecoin Core's higher min tx fee made ~5460-litoshi outputs the practical floor, but the policy dust limit is 546/294 — clamp change to the recipient-script dust limit. CHANGE: fresh own change addr (prefer same address type as inputs); sub-dust change -> fee. RBF: opt-in BIP125 supported (Litecoin tracks Bitcoin Core) — signal via sequence <= 0xFFFFFFFD. SEND-ALL: use_max_amount.
- **Data / memo / tag:** OP_RETURN supported (Litecoin inherits Bitcoin Core's datacarrier policy: 80 bytes data / 1 OP_RETURN output standard). wallet-core output_op_return. EXCHANGES: LTC deposits use a unique ADDRESS (L…/M…/ltc1…); NO memo/destination-tag/OP_RETURN required — compose screen needs no memo field. OP_RETURN optional/advanced only. NO destination tag concept.
- **Other features:** ADDRESS TYPES: P2PKH (L…, prefix 48), P2SH (M…/3…, prefix 50), P2WPKH bech32 (ltc1q…), MWEB (ltcmweb1… — out of scope v1). Validate recipient checksum + that it is LTC (not a BTC bc1…). NO reserve/min-balance. RBF supported (opt-in). SEND-ALL via use_max_amount. MIN RELAY clamp. Litecoin's estimatesmartfee has a KNOWN bug (returns below min relay fee) — do NOT rely on a full node's estimatesmartfee; the litecoinspace Esplora endpoint is the correct source. Compose collects: recipient, amount/max, fee rate (litoshi/vB), optional OP_RETURN, optional RBF. Just-in-time re-fetch UTXO + rate before sign.
- **Tx construction (wallet-core):** wallet-core supports Litecoin natively (segwit-aware). SigningInput = BitcoinSigningInput: hashType = BitcoinScript.hashTypeForCoin(.litecoin) (SIGHASH_ALL, BIP143 for segwit inputs), amount (litoshi), byteFee (litoshi/vB), toAddress (L…/M…/ltc1…), changeAddress (own fresh), utxo[BitcoinUnspentTransaction{outPoint{hash,index,sequence}, amount, script=lockScriptForAddress(coin:.litecoin)}], privateKey[] at sign time. coin_type generally not required (LTC is a first-class coin, CoinType.litecoin). NATIVE MULTI-OUTPUT: extra_outputs. use_max_amount. AnySigner.plan(...,coin:.litecoin) for fee/inputs preview. Encoded raw hex -> broadcast via POST https://litecoinspace.org/api/tx (Esplora broadcast, raw hex body, returns txid). COMPOSE COLLECTS NOW: candidate UTXOs+scripts, fee rate, recipient, amount/max, change addr.

### `dogecoin`
- **Fee model:** FIXED fee-per-kB model, NOT a sat/vB fee market (Dogecoin has NO segwit). Dogecoin Core RECOMMENDED fee = 0.01 DOGE/kB (1,000,000 koinu/kB = 1000 koinu/byte = 1 koinu/byte... precisely 0.01 DOGE per 1000 bytes). MIN RELAY fee = 0.001 DOGE/kB (one-tenth of recommended). Since Dogecoin Core 1.14.5 the fee is computed over EXACT byte size: fee = 0.01/1000 x sizeBytes DOGE (doc example: 192-byte tx = 0.00192 DOGE). So the wallet uses feeRate = 0.01 DOGE/kB (= 1000 koinu/byte) as the NORMAL preset, 0.001 DOGE/kB as a low/min, and a higher multiple for fast. NOTE: live BlockCypher /v1/doge/main returned medium_fee_per_kb=67,716,315 koinu/kB (~0.677 DOGE/kB) on 2026-06-15 — far above the 0.01 floor, reflecting node-reported market rates; the wallet should DEFAULT to the Dogecoin Core 0.01 DOGE/kB recommendation (what wallets actually use and what confirms) and offer BlockCypher's numbers as a 'fast' upper option. Always clamp to >= 0.001 DOGE/kB (hard min relay).
- **Fee fetch RPC:** Two valid approaches. (A) STATIC per the official Dogecoin Core fee recommendation: normal = 0.01 DOGE/kB, slow/min = 0.001 DOGE/kB, fast = 0.05 DOGE/kB — no network call needed, this is the canonical doc-driven rate and what DOGE wallets ship. (B) DYNAMIC oracle (registered primary provider BlockCypher): `GET https://api.blockcypher.com/v1/doge/main` -> {high_fee_per_kb, medium_fee_per_kb, low_fee_per_kb} in koinu/kB (live-verified 2026-06-15: high=270,118,793 / medium=67,716,315 / low=9,379,167). Convert koinu/kB -> koinu/byte by /1000, or DOGE/kB by /1e8. RECOMMENDATION: use the static 0.01 DOGE/kB recommendation as 'normal' (correct & cheap), clamp to the 0.001 hard floor; surface BlockCypher's medium/high as optional faster tiers. Full-node equivalent: estimatefee RPC (DOGE/kB).
- **Gas / compute / weight:** No gas. Raw byte size (NO segwit on DOGE): size ≈ 10 + inputs x 148 (P2PKH) + outputs x 34 (P2PKH). Native-transfer default = 1 input + 2 outputs ≈ 226 bytes -> fee at 0.01 DOGE/kB = 0.01/1000 x 226 = 0.00226 DOGE. Token transfer: N/A (no native tokens; DRC-20 is out of scope). wallet-core computes exact size from byte_fee x transaction_size — but DOGE's byte_fee is koinu/byte: 0.01 DOGE/kB = 1,000,000 koinu / 1000 bytes = 1000 koinu/byte, so byteFee = 1000 (koinu per byte). Recompute after coin selection. CRITICAL: account for the SOFT DUST surcharge (below).
- **Units & decimals:** Smallest unit = koinu (also called 'doge satoshi'). 1 DOGE = 100,000,000 koinu (decimals = 8, wallet-core registry coinId 3). Fee rate expressed as DOGE/kB (0.01) or koinu/byte (1000). Fee total in koinu -> DOGE = /1e8 -> fiat. Decimal math; existing adapter satsPerCoin = 1e8 covers DOGE.
- **UTXO:** YES — UTXO model, P2PKH/P2SH only (NO segwit, NO bech32, NO taproot). Fetch UTXO set from registered primary BlockCypher: `GET https://api.blockcypher.com/v1/doge/main/addrs/{address}?unspentOnly=true&includeScript=true&limit=N` -> {address, balance, n_tx, txrefs:[{tx_hash, tx_output_n(=vout), value(koinu), script(hex locking script!), spent:false, confirmations, confirmed}], hasMore} (live-verified 2026-06-15 — includeScript gives the pkscript inline so NO extra fetch). Page via &before={block_height} when hasMore. The registered dogechain fallback is Cloudflare-gated (no JSON) — BlockCypher is the practical UTXO source. COIN SELECTION: largest-first/BnB or wallet-core planner. DUST: HARD dust limit = 0.001 DOGE (100,000 koinu) — outputs below are non-standard & rejected. SOFT dust limit = 0.01 DOGE (1,000,000 koinu) — any output below 0.01 DOGE REQUIRES an EXTRA 0.01 DOGE added to the fee per such output (else 'too low fee', rejected). ## G2 EVM core

### `ethereum`
- **Fee model:** Two coexisting models; a production compose screen must support BOTH.

(1) EIP-1559 (Type 2 / 0x02) — the DEFAULT for Ethereum mainnet and every modern EVM L1/L2 in this group. Per-gas price = baseFeePerGas (protocol-set, BURNED, adjusts max ±12.5% per block toward a 50%-gas-limit target) + min(maxPriorityFeePerGas, maxFeePerGas − baseFeePerGas). The user/wallet sets two caps: maxPriorityFeePerGas (the tip to the validator) and maxFeePerGas (absolute ceiling per gas). effectiveGasPrice = baseFee + tip (tip capped so total ≤ maxFee). Actual fee charged = gasUsed × effectiveGasPrice; the unused (maxFee − effective) is refunded. Source: EIP-1559 (https://eips.ethereum.org/EIPS/eip-1559) — verified: base fee burned, ±12.5% via BASE_FEE_MAX_CHANGE_DENOMINATOR=8, Type 2 fields chain_id/max_priority_fee_per_gas/max_fee_per_gas/gas_limit/destination/amount/data/access_list.

(2) Legacy (Type 0) — single gasPrice (no base/priority split). Fee = gasUsed × gasPrice. Fallback for any endpoint that doesn't return a base fee (eth_feeHistory empty / pre-London chain). Still valid on mainnet.

Slow/Normal/Fast preset derivation (EIP-1559):
• Fetch baseFeePerGas of the NEXT block from eth_feeHistory (last element of the baseFeePerGas array — it has blockCount+1 entries; the extra is the predicted next-block base fee — confirmed live: 4-block request returned 5 baseFeePerGas entries).
• Fetch reward percentiles via eth_feeHistory rewardPercentiles e.g. [25,50,75] over ~5–20 recent blocks; aggregate (median/avg) each percentile column → slow tip = p25, normal tip = p50, fast tip = p75. (eth_maxPriorityFeePerGas gives a single client-suggested tip as a simpler fallback — live = 0x19a28.)
• maxPriorityFeePerGas[preset] = aggregated percentile tip (floor it to a sane minimum, e.g. ≥1 gwei on mainnet to avoid stuck tx).
• maxFeePerGas[preset] = baseFeeNext × buffer + tip, where buffer covers base-fee rise over the next few blocks (common: 2× base for normal; e.g. slow=1.25×, normal=2×, fast=2.5–3× base, each + its tip). Buffer compensates for the up-to-12.5%/block base-fee climb.
- **Fee fetch RPC:** PRIMARY (EIP-1559): eth_feeHistory — request shape {"method":"eth_feeHistory","params":["0x14","latest",[25,50,75]]} (blockCount hex, newestBlock tag, rewardPercentiles). Returns {oldestBlock, baseFeePerGas[blockCount+1] (wei, last = next-block prediction), gasUsedRatio[], reward[block][percentile] (wei priority fees), baseFeePerBlobGas[], blobGasUsedRatio[]}. Live-verified on https://ethereum-rpc.publicnode.com. publicnode allows 1–1024 blocks per feeHistory call (spec cap; multiplier 1).
SECONDARY tip: eth_maxPriorityFeePerGas — {"params":[]} → single tip in wei (live 0x19a28). Use as tip fallback when feeHistory rewards are empty/zero.
LEGACY fallback: eth_gasPrice — {"params":[]} → single gasPrice wei (live 0x40ee410). Use for Type 0 or when no base fee available.
ALWAYS also fetch: eth_getTransactionCount [addr,"pending"] (nonce — live 0x3), eth_chainId (live 0x1), eth_blockNumber, eth_getBalance [addr,"latest"] (native bal for max-amount).
- **Gas / compute / weight:** eth_estimateGas — request {"method":"eth_estimateGas","params":[{from,to,value,data}]} → gas units (hex). Live-verified:
• Native ETH transfer (value only, no data) = 0x5208 = EXACTLY 21000. Default/floor for a native transfer; NO need to call estimateGas for a plain native send — hardcode 21000 (but still respect a higher result if the `to` is a contract with a fallback).
• ERC-20 transfer (data = transfer(address,uint256) selector 0xa9059cbb + 32-byte to + 32-byte amount, to the token contract, value=0) = live 0xa31a = 41754 for USDT in this case. ERC-20 gas is NOT constant: it depends on the token (USDT/SLOAD warmth, fee-on-transfer logic) and whether the recipient's balance slot is zero→nonzero (cold SSTORE ~20k more) vs nonzero→nonzero. Range ~35k–65k typical; first-time recipients cost more. THEREFORE: ALWAYS call eth_estimateGas live for token transfers (do not hardcode), then pad by ~20–25% safety margin to cover state changes between estimate and inclusion. Defaults if estimate fails: native 21000; ERC-20 fallback 100000 (generous ceiling).
Must pass `from` so estimateGas can read sender balance/allowance; supports a state-override object (fake balance/nonce) for estimating sends the sender can't yet afford.
- **Units & decimals:** Base unit = wei. 1 ETH = 10^18 wei. 1 gwei = 10^9 wei (gas prices are conventionally displayed in gwei). All RPC quantities are hex-encoded big-endian, minimal (no leading zeros, '0x0' for zero) per the Ethereum JSON-RPC QUANTITY encoding rule (https://ethereum.org/en/developers/docs/apis/json-rpc/).
Fee math (use Swift Decimal / arbitrary-precision, NEVER Double):
• EIP-1559 max fee (wei) = gasLimit × maxFeePerGas.
• EIP-1559 expected fee (wei) = gasLimit × (baseFeeNext + tip).
• Legacy fee (wei) = gasLimit × gasPrice.
• Display: feeETH = feeWei / 10^18; feeGwei (for the per-gas price) = priceWei / 10^9.
• Fiat: feeETH × ETH/USD (from the local price store per Rule #27).
Native ETH amount uses 18 decimals; each ERC-20 uses ITS OWN decimals() (USDT/USDC=6, DAI/WETH/most=18, WBTC=8) — MUST read decimals per token, never assume 18. balanceOf via eth_call (selector 0x70a08231) live-verified returns the raw integer in token base units.
- **UTXO:** N/A (account model). Ethereum and all EVM chains are account-based: balance is a single scalar per address (eth_getBalance), nonce is a per-account monotonic counter (eth_getTransactionCount). No UTXO set, no coin selection, no dust limit, no change output, no RBF. The EVM 'replace a pending tx' equivalent is resubmitting the SAME nonce with a higher gas price (must beat the prior tip+maxfee by the client's replacement bump, typically ≥10%) — compose should track the pending nonce so a speed-up reuses it.
- **Data / memo / tag:** EVM has NO memo / destination-tag / OP_RETURN concept. Arbitrary bytes ride in the transaction `data` (input) field (Type 2 field `data`; wallet-core Transfer.data / ContractGeneric.data). For a PLAIN native transfer to an EOA, data is empty (0x). Notes:
• Exchanges do NOT use a memo/tag for ETH/EVM deposits — each user gets a unique deposit ADDRESS (unlike XRP/XLM/Cosmos). So this compose screen must NOT show a memo field for EVM sends — that would be misleading (Rule #2 honesty).
• A token transfer encodes its payload IN data (transfer(address,uint256)); the user-facing 'amount' goes into the ABI-encoded data, NOT the tx `value` (value=0 for ERC-20).
• Optional free-form data on a native send is technically possible but rare and increases gas (16 gas per non-zero byte, 4 per zero byte per EIP-2028); do not surface it as a standard field. If ever added, it raises the gas estimate, so re-run eth_estimateGas with the data included.
- **Other features:** • Max-amount / send-all (NATIVE): maxSend = balance − (gasLimit × maxFeePerGas). MUST reserve the WORST-CASE fee (use maxFeePerGas, not expected), else the tx can't be mined if base fee rises. Recompute whenever the fee preset changes. For ERC-20 send-all: amount = full token balanceOf (eth_call 0x70a08231, live-verified) — token sends are paid in NATIVE ETH for gas, so the token amount is the full balance BUT the wallet must also hold enough native ETH to cover gas (surface 'insufficient ETH for gas' honestly if native balance < fee).
• Nonce management: fetch eth_getTransactionCount(addr,'pending') at compose time (live 0x3); 'pending' includes mempool txs so consecutive sends don't collide. Persist the pending nonce so a 2nd send in the same session increments correctly.
• Replacement / speed-up / cancel: resubmit same nonce, higher fee (cancel = same nonce, value 0, to=self). Compose should expose this via the advanced section.
• Access list (EIP-2930): optional repeated {address, storageKeys} on Type 2 txs; can reduce gas for contract-heavy txs. Advanced-only; default empty.
• EIP-4844 blobs: NOT relevant to user sends (rollup-only); feeHistory returns baseFeePerBlobGas but a wallet send never sets blobs.
• Chain identity: every EVM chain differs ONLY by chainId (eth_chainId, mainnet 0x1) + endpoint + native-coin decimals/symbol. The compose model is otherwise identical across all 12 EVM chains.
• L2 ADDENDUM (Base/Optimism/Arbitrum etc. in the wider EVM set — NOT mainnet): OP-stack L2s charge an additional L1 DATA FEE on top of L2 EIP-1559 gas (totalFee = operatorFee + gasUsed×(baseFee+priorityFee) + l1Fee). Fetch via the GasPriceOracle predeploy 0x420000000000000000000000000000000000000F method getL1Fee(bytes)/getL1FeeUpperBound on the unsigned RLP — source https://docs.optimism.io/stack/transactions/fees. Ethereum MAINNET (this group's canonical chain) has NO L1 data fee; the compose screen must add this component ONLY for OP-stack chains. Arbitrum folds its L1 cost into a higher gasLimit returned by eth_estimateGas, so on Arbitrum the standard estimate already includes it.
- **Tx construction (wallet-core):** wallet-core (TrustWalletCore) Ethereum SigningInput (src/proto/Ethereum.proto, verified): set chainId, nonce, txMode = Enveloped (1) for EIP-1559 [default] or Legacy (0) for Type 0; gasLimit; for EIP-1559 set maxFeePerGas + maxInclusionFeePerGas (the tip) and LEAVE gasPrice empty; for Legacy set gasPrice only. toAddress. privateKey at sign time. Transaction oneof:
• Native send → Transaction.Transfer{amount (wei big-endian), data (empty)}.
• Token send → Transaction.ERC20Transfer{to, amount}; set SigningInput.toAddress = the TOKEN CONTRACT address (wallet-core encodes the transfer() calldata + sets value 0 internally).
• Generic contract call → ContractGeneric{amount, data}.
Also: ERC20Approve, ERC721Transfer, ERC1155Transfer, optional access_list (Enveloped only), eip7702_authorization (SetCode/Type-4, advanced), UserOperation (4337, advanced). TransactionMode enum: Legacy=0, Enveloped=1, UserOp=2, SetCode=4.
FIELDS THIS COMPOSE SCREEN MUST COLLECT NOW so signing works later: recipient address (validated checksum/EIP-55), amount (in token base units via correct decimals), selected fee preset → resolved (maxFeePerGas, maxPriorityFeePerGas) OR gasPrice, gasLimit (from eth_estimateGas + pad), nonce (eth_getTransactionCount pending), chainId, and the asset's contract address+decimals for token sends. Output of signing → eth_sendRawTransaction(signedHex) → returns tx hash (write a pending TransactionRecord per Rule #27 outbox).
- **Doc URLs:** https://ethereum.org/en/developers/docs/apis/json-rpc/; https://github.com/ethereum/execution-apis/blob/main/src/eth/fee_market.yaml; https://eips.ethereum.org/EIPS/eip-1559; https://www.quicknode.com/docs/ethereum/eth_estimateGas; https://raw.githubusercontent.com/trustwallet/wallet-core/master/src/proto/Ethereum.proto; https://docs.optimism.io/stack/transactions/fees; https://www.publicnode.com/; https://eips.ethereum.org/EIPS/eip-2028

## G3 EVM per-chain quirks

### `arbitrum`
- **Fee model:** EIP-1559 (type 0x02, maxFeePerGas/maxPriorityFeePerGas) — DELTA from Ethereum core: the L2 priority/tip market is near-flat; the dominant variable cost is the L1 calldata component, which Arbitrum BAKES INTO THE GAS LIMIT rather than the gas price (so the L2 basefee × gasLimit already covers L1 posting). Effective price = min(maxFeePerGas, baseFee + tip). There is an enforced minimum L2 gas price floor (live: ArbGasInfo.getMinimumGasPrice() = 0x1312d00 = 20,000,000 wei = 0.02 gwei). Presets: slow = baseFee + 0 tip (tip is usually 0 on Arbitrum; ordering is FCFS by the sequencer), normal = baseFee×1.1 + 0.01 gwei tip, fast = baseFee×1.25. Because L1 cost rides in the gas LIMIT, do NOT also add an L1 surcharge to the price.
- **Fee fetch RPC:** L2 price: eth_gasPrice {"method":"eth_gasPrice"} (live → 0x... ~0.01 gwei) and the floor via ArbGasInfo precompile 0x000000000000000000000000000000000000006C: eth_call to 0x6C data=0xf918379a getMinimumGasPrice() (live → 0x1312d00 = 0.02 gwei) and getL1BaseFeeEstimate(). Total/component breakdown: NodeInterface precompile 0x00000000000000000000000000000000000000C8 gasEstimateComponents(to,contractCreation,data) returns (gasEstimate, gasEstimateForL1, baseFee, l1BaseFeeEstimate) and gasEstimateL1Component(...). eth_maxPriorityFeePerGas works (usually returns ~0).
- **Gas / compute / weight:** eth_estimateGas {"to","from","value","data"} — DELTA: a native ETH transfer returns MORE than 21000 because the L1 calldata cost is folded into the returned gas units. Live arbitrum: eth_estimateGas for a 1-wei self-transfer = 0x5339 = 21305 gas. Token (ERC-20) transfer ~ 45k–60k + L1 component. ALWAYS call eth_estimateGas live (the L1 component fluctuates with Ethereum calldata price); never hardcode 21000. Receipt carries gasUsedForL1 (L1 portion expressed in L2 gas units).
- **Units & decimals:** Native = ETH, 18 decimals. Gas price in wei (1 gwei = 1e9 wei). Display fee = gasLimit (incl. L1 component) × effectiveGasPrice / 1e18 ETH, then × ETH/fiat. No separate L1 line item needed for the user since it's inside the gas units.
- **UTXO:** N/A (account model).
- **Data / memo / tag:** None. Standard EVM calldata field (tx `data`); arbitrary hex payload, no protocol memo/tag. Exchanges do NOT require a memo for Arbitrum ETH/token deposits.
- **Other features:** L1 data fee handling is the key delta and is AUTOMATIC (inside gasLimit) — do not double-charge. Native gas token = ETH (Arbitrum One). Sequencer is FCFS so tips rarely matter; surface a single 'Network fee' rather than slow/normal/fast tiers (or keep tiers but vary the gasLimit safety multiplier, not the tip). Max-amount/send-all: subtract gasLimit×maxFeePerGas (which already includes L1) from balance. No account activation, no reserve, no min-balance. Same address format as Ethereum.
- **Tx construction (wallet-core):** wallet-core: id=arbitrum coinId=10042221 chainId=42161 blockchain=Ethereum. Use Ethereum SigningInput with chain_id=42161, tx_mode=Enveloped (EIP-1559), max_fee_per_gas + max_inclusion_fee_per_gas, gas_limit (from eth_estimateGas — MUST use the inflated estimate so L1 cost is covered), transaction.transfer{amount,data}. Compose screen must collect: live gas_limit (eth_estimateGas), maxFeePerGas, tip(usually 0), nonce, value.
- **Doc URLs:** https://docs.arbitrum.io/how-arbitrum-works/gas-fees; https://docs.arbitrum.io/build-decentralized-apps/nodeinterface/reference; https://github.com/trustwallet/wallet-core/blob/master/registry.json; https://raw.githubusercontent.com/trustwallet/wallet-core/master/src/proto/Ethereum.proto

### `optimism`
- **Fee model:** EIP-1559 on L2 (type 0x02) PLUS an additive L1 DATA FEE — this is the core delta. totalFee = l1DataFee + gasUsed×(l2BaseFee + l2PriorityFee) [+ operatorFee post-Isthmus]. The L1 data fee is a SEPARATE, additive line item (unlike Arbitrum, where it's inside the gas limit). Post-Ecotone/Fjord (live: GasPriceOracle.isFjord()=0x01) the L1 fee uses base_fee_scalar (live baseFeeScalar()=0x146b=5227) and blob_base_fee_scalar on the compressed tx size (FastLZ estimator under Fjord). Presets vary the L2 priority fee (slow/normal/fast); the L1 component is the same regardless of tier.
- **Fee fetch RPC:** L2: eth_gasPrice / eth_maxPriorityFeePerGas / eth_getBlockByNumber.baseFeePerGas (live base block baseFeePerGas=0x4c4b40). L1 DATA FEE: eth_call to GasPriceOracle predeploy 0x420000000000000000000000000000000000000F method getL1Fee(bytes rawUnsignedTx)→uint256 (Optimism doc: 'strongly recommended you do NOT implement the L1 fee formula yourself; call getL1Fee'). Pre-sign estimate when no raw tx yet: getL1FeeUpperBound(uint256 unsignedTxSize). Component reads: l1BaseFee() 0x519b4bd3 (live=0x3a65746), baseFeeScalar() 0xc5985918 (live=0x146b), blobBaseFee(), blobBaseFeeScalar(), getL1GasUsed(bytes), isEcotone(), isFjord() 0x4ef6e224 (live=0x01), decimals()=6.
- **Gas / compute / weight:** eth_estimateGas for the L2 execution gas only (native transfer = 21000; ERC-20 transfer ~ 50k–65k). The L1 data fee is ADDED on top via getL1Fee. So total displayed fee = (21000 × l2EffectivePrice) + getL1Fee(serializedTx).
- **Units & decimals:** Native = ETH, 18 decimals; wei/gwei for L2 price. GasPriceOracle.decimals()=6 (scalar precision). L1 fee returned by getL1Fee is already in wei. Display = (l2GasFee + l1DataFee)/1e18 ETH × fiat. Receipt confirms with l1Fee, l1GasUsed, l1GasPrice, l1BaseFeeScalar, l1BlobBaseFee, l1BlobBaseFeeScalar (all verified live on an Optimism receipt).

Operator fee (Isthmus): getOperatorFee(gasUsed) — add if present. Native gas token = ETH. Max-amount/send-all MUST subtract BOTH the L2 gas fee AND the L1 data fee (compute getL1Fee on the would-be tx, then iterate since changing value changes encoded size minimally). Same address format as Ethereum. No reserve/activation.
- **Tx construction (wallet-core):** wallet-core: id=optimism coinId=10000070 chainId=10 blockchain=Ethereum. Ethereum SigningInput, chain_id=10, tx_mode=Enveloped, EIP-1559 fields, gas_limit=21000 native, transfer{amount,data}. The L1 data fee is NOT a wallet-core field — it's purely a fee-DISPLAY/balance-reservation concern computed off the serialized tx; wallet-core signs a standard EIP-1559 tx and the sequencer charges L1 automatically. Compose screen must collect L2 price + L2 gasLimit AND compute getL1Fee for display & send-all.

## G4 Solana

### `solana`
- **Fee model:** Two-part fee: (1) BASE FEE = a fixed 5,000 lamports PER SIGNATURE, statically priced, split 50% burned / 50% to the validator (solana.com/docs/core/fees). A normal SOL transfer or SPL transfer is single-signature => base = 5,000 lamports. (2) OPTIONAL PRIORITIZATION (priority) FEE = ceil(compute_unit_price * compute_unit_limit / 1,000,000) lamports, 100% to the validator. compute_unit_price is in MICRO-lamports per compute unit; 1,000,000 micro-lamports = 1 lamport. CRITICAL: the priority fee is charged on the REQUESTED compute_unit_limit, NOT on actual CUs consumed (solana.com/docs/core/fees/compute-budget), so over-requesting the limit overpays. Slow/Normal/Fast presets are derived ONLY from the priority component (base is always 5,000): call getRecentPrioritizationFees (150-sample cache window), then take e.g. ~25th/50th/75th percentile (or min/median/max) of the non-zero prioritizationFee values as the slow/normal/fast compute_unit_price; apply a sane floor (network is frequently quiet — live test 2026-06-15 returned 150 samples all = 0 micro-lamports, so a 0 preset must still floor to a small non-zero value like 1,000–10,000 micro-lamports for inclusion under load, while Slow may legitimately be 0 when idle). Total displayed fee = 5,000 + ceil(price_microlamports * cu_limit / 1,000,000) lamports. There is NO gasPrice/EIP-1559 and NO sat/vB — this is base + compute-budget priority. A nonce-account (durable nonce) transaction adds an AdvanceNonce instruction but the fee model is unchanged.
- **Fee fetch RPC:** PRIORITY PRICE: getRecentPrioritizationFees — POST {"jsonrpc":"2.0","id":1,"method":"getRecentPrioritizationFees","params":[[<optional writable account base58 addresses, max 128>]]}. Returns array of up to 150 {slot:u64, prioritizationFee:u64} where prioritizationFee = micro-lamports per compute unit observed in that slot; passing the writable accounts the tx will lock (sender, recipient, fee payer, ATA, token mint) yields contention-aware fees, empty array => global samples (solana.com/docs/rpc/http/getrecentprioritizationfees, cache stores up to 150 blocks). EXACT TOTAL FEE FOR A BUILT MESSAGE: getFeeForMessage — POST {"jsonrpc":"2.0","id":1,"method":"getFeeForMessage","params":["<base64 compiled Message (legacy or v0)>",{"commitment":"processed"}]} => result.value = total fee in lamports (base + priority for that exact message), or null if the message's blockhash is no longer valid (solana.com/docs/rpc/http/getfeeformessage; live confirmed value:5000 for a bare transfer). Use getFeeForMessage as the authoritative total once the message (with compute-budget ixs) is assembled. Live tested OK on https://solana-rpc.publicnode.com (apiVersion 4.0.0).
- **Gas / compute / weight:** Solana uses COMPUTE UNITS (CU), not gas. Defaults: 200,000 CU per non-builtin instruction, 3,000 CU per builtin instruction; HARD CAP 1,400,000 CU per transaction (solana.com/docs/core/fees/compute-budget). Set the actual limit with the Compute Budget program (ComputeBudget111111111111111111111111111111) SetComputeUnitLimit instruction, and the price with SetComputeUnitPrice (micro-lamports/CU). Only ONE of each Compute-Budget instruction per tx (duplicate => DuplicateInstruction error). PER-CHAIN DEFAULTS: a bare SOL native transfer (SystemProgram::Transfer) consumes ~150 CU — a safe default limit is ~300–450 CU (or skip SetComputeUnitLimit and let the default 200k stand if not prioritizing). An SPL token transferChecked consumes ~few thousand CU; with ATA creation add more — a safe default limit is ~50,000 CU for a token transfer, ~30,000 for transfer + memo. BEST PRACTICE for an exact limit: build the tx, call simulateTransaction with {sigVerify:false, replaceRecentBlockhash:true} and read value.unitsConsumed, then set SetComputeUnitLimit = unitsConsumed * ~1.1 (10% buffer) (solana.com/docs/rpc/http/simulatetransaction). - **Units & decimals:** Native unit = LAMPORT. 1 SOL = 1,000,000,000 lamports (9 decimals). ## G5 Stellar

### `stellar`
- **Fee model:** FIXED PER-OPERATION inclusion fee (NOT gas/sat-vB). Total inclusion fee = (number of operations in the tx) × (effective base fee for the ledger). Network MINIMUM base fee = 100 stroops per operation (live-confirmed: GET /fee_stats -> last_ledger_base_fee:"100"; GET /ledgers -> base_fee_in_stroops:100). A simple native XLM send = 1 operation => minimum 100 stroops (0.00001 XLM). The `fee` field a wallet sets is the MAX fee it is willing to pay PER OPERATION-equivalent (actually total max bid); the network only charges the effective base fee unless surge pricing is active. SURGE PRICING: kicks in when a ledger exceeds capacity (>1000 ops for classic txs; live ledger_capacity_usage was 0.98-1.15 at test time, i.e. near/over capacity). During surge, txs are ordered by inclusion fee bid and the user pays the minimum inclusion fee of the included set — so a higher bid only buys INCLUSION PRIORITY, never costs more than necessary. PRESETS derived from /fee_stats fee_charged + max_fee percentiles: SLOW = max(100, fee_charged.p50) per op (often 100); NORMAL = fee_charged.p70 or max_fee.p50 (live ~20001); FAST = fee_charged.p90/p95 or max_fee.p90 (live ~250005) — set the tx `fee` to this bid; the user is only CHARGED the effective base fee at inclusion, so a generous max_fee bid is safe. FEE-BUMP TRANSACTIONS (CAP-15): to speed a stuck tx, wrap it in a FeeBumpTransaction with a NEW fee bid that MUST be at least 10× the original inner fee bid; a different `feeSource` account pays. Resource fee (Soroban smart-contract only) is separate and N/A for plain payments/token transfers.
- **Fee fetch RPC:** Horizon REST (NOT JSON-RPC). Primary: `GET https://horizon.stellar.org/fee_stats` — no params, no auth. Live response shape (verified): {"last_ledger":"63033051","last_ledger_base_fee":"100","ledger_capacity_usage":"0.98","fee_charged":{"max","min","mode","p10","p20","p30","p40","p50","p60","p70","p80","p90","p95","p99"},"max_fee":{same percentiles}} — every value is a stringified stroop integer. `fee_charged` = what txs were ACTUALLY charged last ledgers (use for realistic preset display); `max_fee` = what txs BID (use to gauge surge competition). Also `GET /ledgers?order=desc&limit=1` returns base_fee_in_stroops + base_reserve_in_stroops + max_tx_set_size (live: 100 / 5000000 / 1000) for the absolute network minimum. No API key needed for the SDF public Horizon; SDF rate-limits anonymous clients (consider a fallback Horizon provider).
- **Gas / compute / weight:** NO gas/compute estimation for classic operations — fee is purely (op count × base fee). Per-op cost is FIXED at the effective base fee regardless of payload size (classic ops are not metered; only Soroban contract invocations are, and those are N/A here). NATIVE XLM TRANSFER: 1 operation (Payment if dest funded, or CreateAccount if dest unfunded) => min fee 100 stroops. TOKEN (Stellar asset) TRANSFER: still 1 Payment operation => min fee 100 stroops (same as native; the asset is just a field on the Payment op). If the destination lacks a trustline for the asset, the send FAILS (op_no_trust) — it cannot be auto-fixed by the sender. A multi-step send (e.g. CreateAccount + ChangeTrust + Payment) multiplies the base fee by the op count. Max 100 operations per transaction. So the only "estimate" the compose screen needs is: opCount = (dest unfunded ? 1 CreateAccount : 1 Payment); fee = opCount × chosenBaseFeeBid.
- **Units & decimals:** Smallest unit = STROOP = 1×10⁻⁷ XLM (one ten-millionth of a lumen). XLM (and EVERY Stellar asset amount over Horizon) has 7 decimal places. Fees are quoted/charged in STROOPS (integers): 100 stroops = 0.00001 XLM. Amounts in Horizon JSON are decimal strings with up to 7 dp (e.g. "23506.8838472"); in XDR/wallet-core they are int64 stroops (amount_XLM × 10⁷). CONVERSION for display: feeXLM = stroops / 10_000_000; fiat = feeXLM × XLM/fiat price. Money math MUST use Decimal/int64 — never Double — because 7-dp precision and int64 stroop range (max ~9.22e18 stroops = 922,337,203,685 XLM) require exact arithmetic. wallet-core Stellar.proto `fee` is int32 stroops (per-op base-fee bid) and `amount`/`startingBalance` are int64 stroops.
- **UTXO:** N/A (ACCOUNT model). Stellar has no UTXOs, no coin selection, no change outputs, no dust limit in the UTXO sense, no RBF. State is per-account balances keyed by asset. The sendable native balance is NOT the full balance: spendable = total_XLM − minimumBalance − selling_liabilities − (fee for this tx). Fetch the account state with `GET https://horizon.stellar.org/accounts/{accountId}` (live-verified) which returns: `sequence` (REQUIRED for tx construction — the new tx uses sequence+1), `subentry_count`, `num_sponsoring`, `num_sponsored`, `balances[]` (each with asset_type native|credit_alphanum4|credit_alphanum12, asset_code, asset_issuer, balance, limit, buying_liabilities, selling_liabilities, is_authorized), `signers[]`, `thresholds`, `flags`, `data{}`. A 404 from this endpoint means the account is UNFUNDED/non-existent — the compose screen must then require CreateAccount for THIS sender (sender must exist) or, if the DESTINATION 404s, switch the send op to create_account.
- **Data / memo / tag:** MEMO — first-class, attached to the TRANSACTION (one memo per tx, not per op). Five types (XDR Memo union): MEMO_NONE (type 0); MEMO_TEXT (type 1) — ASCII/UTF-8 string, MAX 28 BYTES (byte length, not char count — multibyte UTF-8 counts toward 28); MEMO_ID (type 2) — uint64 (0…18446744073709551615); MEMO_HASH (type 3) — exactly 32 bytes (64 hex chars); MEMO_RETURN (type 4) — exactly 32 bytes (hash of the tx being refunded). EXCHANGES REQUIRE A MEMO: custodial exchanges/wallets pool all user deposits into ONE Stellar account and use the memo (almost always MEMO_ID or MEMO_TEXT) to credit the right user — sending without it can LOSE FUNDS. The compose screen MUST: (1) offer a memo field with a type selector (text/id/hash/return) and enforce the per-type size limit; (2) run the SEP-0029 memo-required check before sending — `GET /accounts/{destId}/data/config.memo_required` (or read the `data` map from the account fetch); if the data entry exists with base64 value "MQ==" (="1") and the user set NO memo, BLOCK the send and prompt for a memo. Live-verified: SDF account returns 404 for that key (not memo-required). SEP-29 check applies to PAYMENT, PATH_PAYMENT_STRICT_SEND/RECEIVE, ACCOUNT_MERGE; SKIP it for CREATE_ACCOUNT (dest doesn't exist yet) and skip if dest is a muxed (M…) account or a memo is already set. MUXED ACCOUNTS (CAP-27, SEP-23, M-prefixed addresses) embed the memo-id IN the address — when sending to an M… address, do NOT also require/attach a separate memo.
- **Other features:** BASE FEE: 100 stroops/op minimum (live last_ledger_base_fee). FEE-BUMP: new bid ≥10× original (speed-up stuck tx). BASE RESERVE: 0.5 XLM (live base_reserve_in_stroops:5000000). MINIMUM BALANCE FORMULA: (2 + subentry_count + num_sponsoring − num_sponsored) × baseReserve. Base account = 2 reserves = 1 XLM. Subentries (+0.5 XLM each): trustlines, offers, signers (beyond master), data entries, pool-share trustlines. So spendable native = balance − minBalance − selling_liabilities − txFee; the compose screen MUST compute this and block sends that would breach the reserve (live SDF acct: subentry_count 6, num_sponsoring/sponsored 0 => minBalance = (2+6)×0.5 = 4 XLM). MAX/SEND-ALL: for native, max = balance − minimumBalance − selling_liabilities − fee (NEVER the full balance — would brick the account); for a non-native asset, max = balance − selling_liabilities (assets don't reserve XLM, but the XLM reserve/fee still must be covered separately). ACCOUNT ACTIVATION: sending native to a NON-EXISTENT (404) destination MUST use CreateAccount with startingBalance ≥ the dest's minimum balance (≥1 XLM for a bare account); using Payment to an unfunded dest fails with op_no_destination. TRUSTLINES: to send a NON-NATIVE asset, the DESTINATION must already hold a trustline for (asset_code, issuer); else op_no_trust — the sender cannot create the dest's trustline; the compose screen should verify the dest's balances[] contains the asset before allowing a token send and surface a clear error otherwise. The SENDER also needs its own trustline (it can't hold the asset without one). ASSET AUTH: issuer may require auth (flags.auth_required) — a dest trustline that isn't is_authorized cannot receive. TIME BOUNDS / preconditions: optional minTime/maxTime (set maxTime to now+N seconds so a stuck tx eventually expires instead of replaying). SEQUENCE NUMBER: must be source.sequence+1 (fetch fresh just before signing — a stale sequence => tx_bad_seq). NO native-token gas concept, NO priority tip beyond the fee bid.
- **Tx construction (wallet-core):** wallet-core (WalletCore) SUPPORTS Stellar: TWCoinTypeStellar, SigningInput in Stellar.proto (verified upstream). Fields the COMPOSE screen must collect NOW so signing works later: `account` (source/sender G… address), `passphrase` (network passphrase string — PUBLIC: "Public Global Stellar Network ; September 2015", TESTNET: "Test SDF Network ; September 2015"), `fee` (int32, per-op base-fee bid in stroops, from /fee_stats preset), `sequence` (int64 = current account sequence + 1, fetch fresh), one operation oneof — op_payment {destination, amount(int64 stroops), asset} OR op_create_account {destination, amount=startingBalance} OR op_change_trust {asset} OR op_create_claimable_balance / op_claim_claimable_balance — and one memo oneof: memo_void / memo_text(string ≤28 bytes) / memo_id(uint64) / memo_hash(32 bytes) / memo_return_hash(32 bytes). Optional time_bounds(int64). SigningOutput returns `signature` = the base64 XDR transaction envelope ready to POST to Horizon `POST /transactions?tx={xdr}`. NO native multi-output: wallet-core's Stellar signer encodes exactly ONE operation per SigningInput (count hardcoded to 1) — to send to multiple recipients or do CreateAccount+ChangeTrust+Payment atomically you must build the multi-op XDR yourself (custom XDR), wallet-core won't. CRITICAL LIMITATION (verified in Signer.cpp encodeAsset): the upstream wallet-core Stellar signer supports ONLY native + credit_alphanum4 assets (asset codes ≤4 chars) — credit_alphanum12 (5-12 char codes like "USDCAllow") is NOT encodable via wallet-core; for alphanum12 token sends the compose/sign layer must build the XDR manually (or patch wallet-core). The compose screen MUST therefore detect asset_type==credit_alphanum12 and route it to a custom XDR path, not the wallet-core SigningInput path. Set up the SEP-29 memo gate, reserve/min-balance gate, trustline check, and create_account-vs-payment branch at COMPOSE time so the later sign step has all inputs.

## G6 Sui

### `sui`
- **Fee model:** Sui has a DECOUPLED two-part fee model (NOT EVM gas-price, NOT sat/vB, NOT fixed). total_fee = computation_fee + storage_fee - storage_rebate, where computation_fee = computation_units * reference_gas_price and storage_fee = storage_units * storage_price (storage_price linear at 100 storage-units/byte). Two values the wallet must set per tx: (1) GAS PRICE = the per-computation-unit price in MIST. The wallet MUST set this >= the epoch Reference Gas Price (RGP); a higher value is a TIP for priority: ComputationPrice = ReferencePrice + Tip. (2) GAS BUDGET = the max MIST the tx may spend; it must satisfy gas_budget >= max(computation_fees, total_gas_fees). Min budget 2,000 MIST (0.000002 SUI); MAX budget 50,000,000,000 MIST (50 SUI) (overflow/DoS guard). Computation is bucketed into coarse tiers (smallest bucket = 1,000 units, charged 100,000 MIST at RGP=100; largest ~5,000,000). LIVE-VERIFIED: a native SUI transfer dry-ran to computationCost 100,000 + storageCost 1,976,000 - storageRebate 978,120 = NET 1,097,880 MIST = 0.00109788 SUI (RGP=100). Slow/Normal/Fast presets are NOT derived from a fee-history percentile like EVM; instead: Normal gasPrice = RGP (suix_getReferenceGasPrice); Fast = RGP * 1.1-1.5 (additive tip for priority during congestion); Slow = RGP exactly (cannot go below RGP or the tx is rejected). The gas BUDGET (not price) is what should be auto-sized from a dry run (see gasEstimate).
- **Fee fetch RPC:** TWO live calls. (1) Reference gas price (the price floor/normal): suix_getReferenceGasPrice -> {"jsonrpc":"2.0","id":1,"method":"suix_getReferenceGasPrice","params":[]} -> result is a BigInt<u64> string of MIST, LIVE-CONFIRMED "100". (2) Budget + actual fee breakdown via DRY RUN: sui_dryRunTransactionBlock -> {"jsonrpc":"2.0","id":1,"method":"sui_dryRunTransactionBlock","params":["<base64 BCS TransactionData>"]} -> result.effects.gasUsed = GasCostSummary {computationCost, storageCost, storageRebate, nonRefundableStorageFee} (all BigInt<u64> MIST strings). result also carries balanceChanges[] (signed per-coinType deltas) and effects.status. LIVE-CONFIRMED gasUsed = {computationCost:"100000", storageCost:"1976000", storageRebate:"978120", nonRefundableStorageFee:"9880"}. Note: DryRunTransactionBlockResponse now also has an optional suggestedGasPrice field (null in my test). Alternative for estimation without a full TransactionData: sui_devInspectTransactionBlock(sender_address, tx_bytes=BCS TransactionKind, gas_price?, epoch?, additional_args{gas_budget,gas_objects,gas_sponsor,skip_checks}) -> DevInspectResults (gas usage calculated, gas not charged).
- **Gas / compute / weight:** Sui has NO eth_estimateGas-style call returning a single 'gas limit'; instead the wallet sizes the GAS BUDGET from a DRY RUN. Recommended flow: build the unsigned TransactionData (PTB) with a generous provisional budget, call sui_dryRunTransactionBlock, read effects.gasUsed, then set the real gas_budget = computationCost + storageCost + safety_margin (do NOT subtract storageRebate when budgeting since the rebate is applied post-execution; budgeting on the gross protects against under-funding). LIVE-VERIFIED for a native SUI transfer: gross = computationCost 100,000 + storageCost 1,976,000 = 2,076,000 MIST; net charged 1,097,880 MIST. PER-CHAIN DEFAULTS (when skipping a dry run for instant UX, then refine via dry run before sign): native SUI transfer budget default ~3,000,000-4,000,000 MIST (0.003-0.004 SUI) is safe headroom; fungible-token (Coin<T>) transfer default ~4,000,000-5,000,000 MIST because a non-SUI transfer needs a separate SUI gas coin + splitCoins + transferObjects, adding storage for the new coin object. Computation for a simple transfer lands in the smallest bucket (1,000 units = 100,000 MIST at RGP=100). gas_budget must always be within [2,000 ; 50,000,000,000] MIST.
- **Units & decimals:** Fee unit = MIST (the smallest unit of SUI). 1 SUI = 1,000,000,000 MIST = 10^9 MIST (SUI has 9 decimals; confirmed in official framework source sui::sui MIST_PER_SUI = 1_000_000_000 and wallet-core registry decimals=9). RGP, gas_budget, computationCost, storageCost, storageRebate, coin.balance, and amounts are ALL in MIST (BigInt strings over JSON-RPC). To display: native amount in SUI = MIST / 10^9 (use Swift Decimal, never Double, for money math). Fiat = (MIST/10^9) * SUI_price. Fungible tokens have their own decimals from the coin metadata (e.g. USDC on Sui = 6) fetched via suix_getCoinMetadata; never assume 9 for tokens.
- **UTXO:** HYBRID object model, NOT classic UTXO but with UTXO-like coin selection for the CLASSIC path AND a new account-balance path. CLASSIC (Coin<T> object) model: an address's balance of a coin type is spread across many separate Coin<T> objects (each an owned object with its own objectId/version/digest). To send, the wallet fetches the coin set and selects/splits/merges. Fetch the coin set: suix_getCoins(owner, coin_type="0x2::sui::SUI"|token type, cursor, limit) or suix_getAllCoins(owner, cursor, limit) -> CoinPage{data:[Coin{coinType, coinObjectId, version, digest, balance, previousTransaction}], nextCursor, hasNextPage} (paginate via cursor until hasNextPage=false). COIN SELECTION: pick coin object(s) whose summed balance >= amount + budget; if one coin suffices, split it (PTB splitCoins) for the exact send amount and transfer the split, change stays in the source coin; if multiple needed, MERGE them (mergeCoins) or use GAS SMASHING (pass multiple coins as gas payment; up to 256 coins per PTB; the network smashes them into the first coin, change+rebate credited to the first gas coin). NO DUST LIMIT on Coin objects (a coin can hold any balance incl. 0; tiny coins just cost storage). NO RBF (Sui has no mempool fee-replacement; finality is fast via consensus/fast-path). NEW ADDRESS-BALANCES path (SIP-58, live on mainnet 2026-05-20): an address can hold a single canonical Balance per coin type (getBalance.fundsInAddressBalance) that needs NO coin selection at all — leave gas_data.payment empty to pay gas directly from the SUI address balance, and use coinWithBalance to draw the send amount from the address balance automatically. getBalance.totalBalance includes BOTH coin objects AND fundsInAddressBalance; coin-object-only balance = totalBalance - fundsInAddressBalance.
- **Data / memo / tag:** None. Sui has NO memo, NO destination tag, NO OP_RETURN, NO comment, NO arbitrary-data field on a coin transfer. Confirmed: the wallet-core Sui.proto has zero memo/data fields across Pay/PaySui/PayAllSui/TransferObject; the object/PTB model has no transfer-note primitive. A SUI deposit to a centralized exchange uses the plain 32-byte address ONLY — no tag is ever required (memos/tags are an account-tag-chain concept: XRP destination tag, Cosmos/Stellar/TON memo — Sui is not one of those). The compose screen MUST NOT show a memo/tag field for Sui. (If an app ever needed an on-chain note it would have to be a custom Move call with a string arg in a PTB, which is out of scope for a standard transfer and never exchange-required.)
- **Other features:** REFERENCE GAS PRICE: fetch via suix_getReferenceGasPrice; set tx gasPrice >= RGP (Normal=RGP; Fast=RGP+tip). GAS BUDGET: min 2,000 MIST, max 50 SUI; auto-size via dry run. GAS COIN SELECTION/MERGE (gas smashing): for native SUI sends, the gas coin and the value being sent can be the SAME coin pool; pass multiple SUI coins as gas payment and the network smashes them into one (change + storage rebate credited to the FIRST gas coin); max 256 gas coins/PTB. For TOKEN (non-SUI) sends, a SEPARATE SUI gas coin is REQUIRED (gas is always paid in SUI). OBJECT-MODEL IMPLICATIONS FOR COMPOSE: the compose screen must, at amount-entry time, know the coin set so it can warn if the user has many tiny coins / needs a merge; for token sends it must ensure a SUI gas coin exists (otherwise 'no gas coin' error even with enough token balance). SEND-ALL / MAX: use PayAllSui (transfers the ENTIRE balance of the provided input_coins to one recipient, gas deducted from the smashed coin) — this is the correct max-send primitive for native SUI; do NOT compute amount = balance - fee manually then call paySui (rounding/rebate makes that fragile). For token send-all, transfer all token Coin objects and pay gas from a separate SUI coin. ADDRESS BALANCES (SIP-58, mainnet 2026-05-20): if used, gas_data.payment can be empty (gas drawn from SUI address balance) and send amount drawn via coinWithBalance — eliminates coin selection; but a recipient wallet without address-balance support won't see funds sent that way, so compose should prefer the classic Coin path for exchange/unknown recipients. SPONSORED TRANSACTIONS: Sui supports a separate gas sponsor (gas_data.owner != sender) so a third party pays gas; wallet-core Sui SigningInput exposes signer + gas fields to support this. ACCOUNT ACTIVATION: NONE — a Sui address needs no on-chain activation/min-balance/existential-deposit to receive; receiving SUI just creates a Coin object owned by the address. NO reserve/min-balance like XRP/Stellar. EPOCH: RGP can change per epoch (~24h) — refetch RGP before each compose, do not cache across epochs. lockedBalance field exists in Balance (staked/locked) but is {} for normal coins.
- **Tx construction (wallet-core):** wallet-core (WalletCore) DOES support Sui (registry coinId 784, curve ed25519, addr = 32-byte = 0x + 64 hex, derivation m/44'/784'/0'/0'/0'). SigningInput (Sui.proto) fields the compose step must collect NOW so signing works later: signer (sender address string), gas_budget (uint64 MIST), reference_gas_price (uint64 MIST from suix_getReferenceGasPrice), private_key (at sign time), and a transaction_payload oneof. For NATIVE SUI transfer use PaySui{ input_coins: repeated ObjectRef, recipients: repeated string, amounts: repeated uint64 (MIST) }. For MAX/send-all native use PayAllSui{ input_coins: repeated ObjectRef, recipient: string }. For TOKEN (Coin<T>) transfer use Pay{ input_coins: repeated ObjectRef, recipients, amounts, gas: ObjectRef (the SEPARATE SUI gas coin) }. For a single object use TransferObject{ object: ObjectRef, recipient, gas: ObjectRef }. ObjectRef = { object_id: string(hex), version: uint64, object_digest: string(base58) } — so the compose step MUST capture, for every selected input coin AND the gas coin, the live (objectId, version, digest) triple from suix_getCoins (these are consensus-fresh; re-fetch immediately before signing because version bumps on every mutation, else the tx is stale/rejected). wallet-core natively supports MULTI-OUTPUT (PaySui/Pay take repeated recipients+amounts) so batch sends are possible. wallet-core builds the BCS TransactionData and returns unsigned_tx + signature; broadcast via sui_executeTransactionBlock(tx_bytes base64, signatures=[flag||sig||pubkey base64], options, request_type). For PTBs not expressible via the high-level payloads (splitCoins/mergeCoins/address-balance coinWithBalance), use SignDirect{ unsigned_tx_msg: base64 BCS } — build the PTB with the Sui SDK/BCS and pass the bytes to wallet-core for ed25519 signing. NOTE: wallet-core's high-level Sui payloads currently model the CLASSIC explicit-coin path (input_coins/gas ObjectRef); the empty-payment ADDRESS-BALANCE gas path is not expressible via PaySui/Pay and would need SignDirect with an SDK-built PTB.

## G7 TON

### `ton`
- **Fee model:** TON has NO user-set gas price and NO fee market (validators set all prices via network config; user cannot override). A transfer fee is the SUM of four phase-based components, all in nanoton: (1) IMPORT/in_fwd_fee — paid for the incoming EXTERNAL message that carries the signed transfer into the wallet contract (scales with the external message size); (2) STORAGE_FEE — paid for keeping the wallet's code+data stored since the last tx (tiny, ~a few thousand nanoton for a wallet); (3) GAS/COMPUTE FEE — TVM execution of the wallet contract (signature check + building the outgoing message), BaseChain price = 400 nanoton per gas unit, with flat_gas_limit/flat_gas_price for the minimum; (4) FORWARD FEE (fwd_fee) + ACTION FEE — for each outbound INTERNAL message the wallet emits to the recipient. Forward fee formula (basechain, live config 24): msg_fwd_fees = lump_price + ceil((bit_price*msg.bits + cell_price*msg.cells) / 2^16); action_fee = floor(msg_fwd_fees * first_frac / 2^16) (~1/3); fwd_fee = msg_fwd_fees - action_fee. NOTE: the FORWARD/ACTION fee is deducted from the AMOUNT ATTACHED to the outgoing message, NOT added on top — unless send-mode flag +1 (PAY_FEES_SEPARATELY) is set, in which case fees are taken separately from the wallet balance (this is the standard wallet behavior, mode 3 = 1|2). There is no slow/normal/fast tier in the protocol — fees are deterministic given the message size and execution. A wallet derives a single 'network fee' estimate (optionally +20% headroom). For a plain TON transfer this lands around 0.0045-0.006 TON in practice; wallets typically reserve ~0.01-0.05 TON of headroom and, for send-all, subtract the estimated fee from the amount.
- **Fee fetch RPC:** PRIMARY (live-verified 2026-06-15): toncenter v2 estimateFee. POST https://toncenter.com/api/v2/estimateFee with JSON body {"address":"<wallet EQ.. address>","body":"<base64 BoC of the in-message body>","init_code":"<base64 opt, for not-yet-deployed wallet>","init_data":"<base64 opt>","ignore_chksig":true}. Live response confirmed: {"ok":true,"result":{"@type":"query.fees","source_fees":{"@type":"fees","in_fwd_fee":66667,"storage_fee":8789,"gas_fee":0,"fwd_fee":0},"destination_fees":[]}}. TOTAL fee = in_fwd_fee + storage_fee + gas_fee + fwd_fee (all nanoton). Pass ignore_chksig:true so the estimate succeeds without a real signature. The @ton/ton SDK wraps this as TonClient.estimateExternalMessageFee(). The config prices that drive the formula are fetched via GET https://toncenter.com/api/v2/getConfigParam?param=N — live-verified: param=24 (basechain msg_forward_prices: lump_price=10000000, bit_price=655360000 i.e. 10000/bit, cell_price=65536000000 i.e. 1000000/cell, first_frac=21845≈1/3), param=21 (basechain gas: 1 gas=400 nanoton, gas_limit=6667, gas_credit=10000), param=18 (storage prices). Address bounceable/non-bounceable + raw form via GET /api/v2/detectAddress?address=.. (live-verified). Alt providers (same shape): toncenter v3 /api/v3/estimateFee, tonapi.io POST /v2/wallet/emulate (confirmed exists, requires boc), dRPC/Chainstack/Ankr mirror the v2 estimateFee.
- **Gas / compute / weight:** There is NO eth_estimateGas equivalent the wallet must call — TVM gas is determined by execution and estimateFee returns the resulting gas_fee directly. For COMPOSE, the wallet does not set a gas limit; it sets the send-mode and the attached TON amount. Practical guidance: (a) NATIVE TON transfer — gas is ~3000-10000 gas units (signature verify + send one internal message); at 400 nanoton/gas that is ~0.0012-0.004 TON, plus import (~0.0007 TON) + storage (~0.00001 TON) + outbound forward (~0.0004 TON) → total ≈ 0.0045-0.006 TON. External-message gas_credit is 10,000 gas units (the contract computes for free up to this then pays). (b) JETTON (token) transfer — the wallet sends an internal message to ITS OWN jetton wallet which then forwards; this costs MORE: attach ~0.05 TON total to the outgoing message (TON Foundation recommendation), with forward_ton_amount set to 1 nanoton (to trigger the transfer_notification) — excess is returned to response_destination. Always call estimateFee on the fully-built body for the real number; for token transfers the user attaches TON (not the token) for gas, so the compose screen must surface 'network fee in TON' separately and ensure the wallet's TON balance covers it.
- **Units & decimals:** Native unit: TON. Smallest unit: nanoton (also called nanoTON / 'gram' historically). 1 TON = 1,000,000,000 nanoton = 10^9. DECIMALS = 9. ALL RPC fields are in nanoton as integers (or as decimal STRINGS for balance, e.g. getAddressBalance returns "1592537943871182" = 1,592,537.94 TON). estimateFee fields (in_fwd_fee/storage_fee/gas_fee/fwd_fee) are nanoton integers. Config price internal accounting uses nanoton * 2^16 for precision (divide by 65536 in the formulas). Conversion to display: TON = nanoton / 10^9 (use Decimal, never Double). Jettons (tokens) have their OWN decimals from the jetton metadata (commonly 9, but USDT-jetton on TON = 6) — fetch via getTokenData / get_jetton_data, never assume.
- **UTXO:** N/A (account model). TON is an account-based actor/smart-contract model — every wallet IS a smart contract. There is no UTXO set, no coin selection, no change, no dust limit, no RBF. The 'account state' is the wallet contract's code+data cell. Compose-relevant analogues instead of UTXOs: (1) seqno — the wallet's monotonic message counter, the replay-protection nonce; MUST be fetched fresh and embedded in the signed external message (stale seqno → tx rejected with exit code 33). Fetch via getWalletInformation (live-verified field seqno=343) OR runGetMethod method="seqno". (2) account_state — 'active' / 'uninitialized' / 'frozen'; an uninitialized wallet must include state_init (init_code+init_data) on its first tx (the wallet-core SignerTest deploys with seqno 0 + state_init). (3) balance must cover amount + fee; for send-all use send-mode 128.
- **Data / memo / tag:** YES — TON's destination-tag/memo equivalent is the TEXT COMMENT, and exchanges REQUIRE it for deposits. Format: the internal message BODY begins with a 32-bit zero opcode 0x00000000 (32 zero bits) followed by the comment as UTF-8 bytes. In wallet-core this is simply transfer.set_comment("<text>") (proto field 4) — wallet-core builds the 0x00000000-prefixed snake cell automatically. In raw cell terms: beginCell().storeUint(0,32).storeStringTail("<utf8>"). MAX SIZE: a single cell holds 1023 bits ≈ 127 bytes; after the 4-byte opcode, ~123 bytes fit in the root cell. Longer comments use SNAKE encoding (chained ^Cell references) and can be arbitrarily long, but exchange deposit memos are short. EXCHANGE-REQUIRED: Binance, OKX, MEXC, KuCoin, Bybit etc. assign a shared TON deposit address + a unique numeric/alphanumeric MEMO (a.k.a. 'comment' / 'tag'); sending WITHOUT the memo (or wrong memo) loses or delays the deposit (confirmed by multiple exchange help pages + user-loss reports). The compose screen MUST expose a 'Comment / Memo (required by some exchanges)' field, warn when sending to a known-CEX address with no memo, and forbid binary 0xff-prefixed comments for the user field (those are not displayed). For JETTON transfers the comment goes inside forward_payload (also 0x00000000 + UTF-8) and requires forward_ton_amount>0 to deliver the notification carrying it.
- **Other features:** BOUNCEABLE vs NON-BOUNCEABLE address (critical): EQ.. = bounceable (mainnet), UQ.. = non-bounceable (mainnet); kQ../0Q.. = testnet. Sending to an UNINITIALIZED wallet with bounce=true causes funds to bounce back; wallets/exchanges expect non-bounceable (UQ) for plain deposits to fresh wallets. detectAddress (live-verified) returns both forms + raw 0:hex; compose must let the user paste either and resolve the correct bounce flag (wallet-core Transfer.bounceable, default true; for normal user-to-user TON sends to wallets, send NON-bounceable / bounce=false so funds aren't lost to an uninitialized recipient). WORKCHAIN: wallets live in workchain 0 (basechain, '0:'); masterchain is -1 ('-1:', higher fees, rarely used by wallets) — validate workchain. SEND MODES (wallet-core SendMode enum, combine as bitmask): DEFAULT=0, PAY_FEES_SEPARATELY=+1, IGNORE_ACTION_PHASE_ERRORS=+2 (mode 3 = 1|2 is the standard wallet transfer), +16 bounce-on-fail (NEVER on external/wallet msgs), DESTROY_ON_ZERO_BALANCE=+32, ATTACH_ALL_INBOUND_MESSAGE_VALUE=+64, ATTACH_ALL_CONTRACT_BALANCE=+128. SEND-ALL: mode 128 carries the entire wallet balance (fee taken from it); mode 128+32 also destroys/empties the account — wallets use 128 for max-send and subtract nothing extra. valid_until / EXPIRE: every external message carries valid_until (UNIX seconds); wallet-core set_expire_at, default now()+60. Wallet-core errors if expired (exit 35). WALLET VERSIONS: V3R1/V3R2/V4R2/V5R1 (wallet-core enum 0..3); the address+seqno+wallet_id depend on version — the live test wallet was 'wallet v3 r2' wallet_id=698983191. V4/V5 support plugins; V5R1 enforces IGNORE_ERRORS and supports gasless/sponsored fees. JETTON transfer (TEP-74 op 0x0f8a7ea5): query_id, amount(VarUInt16), destination(owner TON addr), response_destination(excesses return — set to sender), custom_payload(Maybe ^Cell), forward_ton_amount(VarUInt16 — >0 to notify, recommended 1 nanoton), forward_payload(Either Cell ^Cell — carries the comment). The transfer is sent to the SENDER's jetton wallet, found via get_wallet_address(owner) get-method on the jetton master; attach ~0.05 TON gas. NO MULTI-OUTPUT in one internal message, but up to 4 internal messages per external message (4-ref cell limit) — wallet-core SigningInput.messages caps at 4. ihr_fee/IHR routing exists but wallets set it 0. extra-currencies field exists in the message value but is unused by normal transfers.
- **Tx construction (wallet-core):** wallet-core (WalletCore) FULLY supports TON. SigningInput (TheOpenNetwork.proto): private_key(1, 32B) OR public_key(2, for external signing), messages(3, repeated Transfer, MAX 4), sequence_number(4, uint32 = the seqno fetched live; 0 for first deploy), expire_at(5, uint32 UNIX; default now()+60), wallet_version(6, enum WALLET_V3_R1=0/V3_R2=1/V4_R2=2/V5_R1=3). Transfer message: dest(1, string — recipient address), amount(2, bytes = uint128 big-endian nanoton, built via store(uint256)), mode(3, uint32 send-mode bitmask), comment(4, string — auto-wrapped as 0x00000000+UTF-8 snake), bounceable(5, bool), state_init(6, base64 1-cell BoC — for deploying an uninitialized wallet), jetton_transfer(7, JettonTransfer), custom_payload(8, base64). JettonTransfer: query_id(1), jetton_amount(2, uint128 BE), to_owner(3), response_address(4), forward_amount(5, uint128 BE nanoton), custom_payload(6). SigningOutput: encoded(1, base64 signed external-message BoC — broadcast as-is to POST /api/v2/sendBoc {"boc":..} or sendBocReturnHash for the tx hash), hash(2, the tx/cell hash for tracking), error(3)/error_message(4). FIELDS THE COMPOSE SCREEN MUST COLLECT NOW so signing works later: (1) recipient address + resolved bounceable flag (default false for user wallets), (2) amount in nanoton (Decimal), (3) comment/memo string (exchange-required), (4) send-mode (3 normal, 128 for send-all), (5) wallet_version of the user's own wallet (determines seqno/address), (6) seqno — fetched JUST BEFORE signing (getWalletInformation), (7) expire_at, (8) for tokens: jetton master address → resolve sender jetton wallet via get_wallet_address, plus forward_amount (1 nanoton) and response_address (sender). The address+public key derive from the private key via wallet-core; the wallet's on-chain address is version-dependent and must match the funded account.

## G8 XRP / Ripple

### `ripple`
- **Fee model:** FIXED per-transaction cost in DROPS that is destroyed (burned), NOT a gas model. Every transaction carries a `Fee` field = an exact integer drop amount the sender commits to before signing. The minimum/reference cost for a standard Payment is the BASE FEE = 10 drops (0.00001 XRP), confirmed live: `server_info.validated_ledger.base_fee_xrp = 1E-5`. Under network load the cost SCALES UP by a load_factor: current cost (drops) = (base_fee * load_factor) / load_base. There are TWO thresholds: the OPEN-LEDGER cost (meet it -> included in the current ledger immediately) and the QUEUE/local-load cost (meet only this -> transaction is queued for a later ledger). Special non-Payment costs (not needed for a plain send but must be known): SetRegularKey reset = 0; multisigned = 10 * (1 + num_signatures); AccountDelete = 2,000,000 drops; AMMCreate = 2,000,000 drops. Slow/Normal/Fast presets derive from the live `fee` method: Slow = drops.minimum_fee (queue, cheapest), Normal = max(drops.open_ledger_fee, base_fee) (immediate inclusion at current load), Fast = a multiple of drops.open_ledger_fee or drops.median_fee for congested ledgers (e.g. Fast = drops.median_fee when median_fee > open_ledger_fee). At idle all three collapse to 10 drops (confirmed live: base_fee=10, open_ledger_fee=10, minimum_fee=10, median_fee=5000). Always cap the fee to a sane max (e.g. a few thousand drops) so a congestion spike never silently overcharges; the user sees the chosen drop amount + XRP + fiat.
- **Fee fetch RPC:** PRIMARY: `fee` method (FeeEscalation amendment). JSON-RPC POST body: {"method":"fee","params":[{}]}. WebSocket: {"id":1,"command":"fee"}. Live response (verified xrplcluster.com 2026-06-14): result.drops.base_fee, result.drops.open_ledger_fee, result.drops.median_fee, result.drops.minimum_fee (all strings of drops); result.current_ledger_size, result.current_queue_size, result.expected_ledger_size, result.max_queue_size; result.levels.{median_level,minimum_level,open_ledger_level,reference_level}. SECONDARY (also gives reserves in one call): `server_info` -> result.info.validated_ledger.{base_fee_xrp, reserve_base_xrp, reserve_inc_xrp} and result.info.load_factor. JSON-RPC body: {"method":"server_info","params":[{}]}. For integer math prefer `server_state` (load_base + load_factor as integers). Compose screen should poll `fee` (cheap) for the live presets and `server_info` once for reserves; both are public, no auth.
- **Gas / compute / weight:** NO gas/compute/weight model — XRP fees are fixed-by-transaction-type, not metered by execution. There is nothing to estimate like eth_estimateGas. For a NATIVE XRP transfer (Payment with XRP Amount) the cost is exactly 1x the reference cost = base_fee (10 drops at idle), scaled by load_factor. For a TOKEN (issued-currency / IOU) transfer the on-ledger transaction type is still Payment and the cost is still 1x the reference cost (10 drops base) — issued-currency value moves over trust lines and does NOT change the burn fee (it MAY require paths/SendMax and a transfer fee charged by the issuer, but that is the issuer's TransferRate, not the network burn fee). So: native default fee = base_fee*load_factor; token-transfer default fee = same base_fee*load_factor. The only multipliers are multisign (per-signer) and the special high-cost types (AccountDelete/AMMCreate) which the compose screen does not emit.
- **Units & decimals:** Native unit = XRP; smallest unit = DROP. 1 XRP = 1,000,000 drops (10^6, exactly 6 decimal places). The Fee field and all XRP Amounts in transactions are STRINGS of integer drops (e.g. "10" = 10 drops = 0.00001 XRP; "56760625556" drops = 56760.625556 XRP, seen live). Display: drops/1_000_000 -> XRP with up to 6 dp; XRP * price -> fiat. NEVER use floating point for drops — use integer/Decimal. server_info returns reserve_base_xrp/reserve_inc_xrp in XRP (e.g. 1E0, 2E-1) which must be converted to drops (1 XRP=1e6 drops) for balance math. Issued-currency (token) amounts are decimal strings with up to 15 significant digits and live in an object {currency, value, issuer}, NOT drops.
- **UTXO:** N/A (account model). XRP Ledger is account-based, not UTXO. No UTXO set to fetch, no coin selection, no dust limit, no change output, no RBF. Balance + next sequence come from `account_info` (JSON-RPC: {"method":"account_info","params":[{"account":"r...","ledger_index":"current","queue":true}]}). Live response gives account_data.Balance (drops string), account_data.Sequence (next tx sequence — verified live Sequence:44196), account_data.OwnerCount (owned objects, for reserve math), account_data.Flags. Replacement of a stuck tx = resubmit a new tx with the SAME Sequence and a higher Fee (fee-escalation queue), which is XRP's analogue of RBF, not Bitcoin RBF.
- **Data / memo / tag:** THREE distinct attachable data features, all compose-relevant:
1) DestinationTag (uint32, 0..4294967295): top-level Payment field. The exchange/custodial routing id. MOST CENTRALIZED EXCHANGES REQUIRE IT — a deposit without the correct tag is credited to the wrong customer or lost. The destination account can ENFORCE it via the lsfRequireDestTag account flag (`requireDestinationTag:true`) — VERIFIED LIVE on rHb9CJAWyB4rj91VRWn96DkukG4bwdtyTh (also disallowIncomingXRP:true). Compose MUST collect a destination tag, validate it as a uint32, and SHOULD pre-check the recipient's `requireDestinationTag` flag (from account_info account_flags) and force the field when set. This is the single most important non-amount field on XRP sends.
2) SourceTag (uint32): top-level common field — identifies the sender / reason on the sender's side (e.g. for return-to-sender routing). Optional; expose as an advanced field.
3) Memos (array, common field): each element is {Memo:{MemoData,MemoType,MemoFormat}} where every sub-field is a HEX-ENCODED string. Total Memos field <= 1 KB serialized. Optional and rarely required by exchanges (they use DestinationTag, not Memos) — expose under advanced. Also InvoiceID (256-bit / 64-hex-char) is an optional structured payment-reason id.
- **Other features:** RESERVES / MINIMUM BALANCE (must drive max-send + activation): Base reserve = 1 XRP, Owner reserve = 0.2 XRP per owned object — VERIFIED LIVE (reserve_base_xrp:1E0, reserve_inc_xrp:2E-1). Minimum (locked) balance = base_reserve + OwnerCount*owner_reserve. SPENDABLE = Balance - minimum_reserve - Fee. An account can never go below its reserve by an ordinary send; the compose screen MUST compute spendable from account_info (Balance, OwnerCount) + live reserves and cap max-amount accordingly. ACCOUNT ACTIVATION: sending to a NOT-YET-EXISTING destination requires the first payment deliver AT LEAST the base reserve (>=1 XRP) or it fails (actNotFound/tecNO_DST_INSUF_XRP) — compose must warn when funding a brand-new account that the amount must clear the base reserve. SEND-ALL vs RESERVE: 'Max'/'Send all' = Balance - reserve_total - Fee (never the full balance; the reserve is permanently non-spendable while the account stays open). The ONLY way to recover the reserve is AccountDelete (cost 2,000,000 drops, leaves <=~0 and requires no owned objects + sequence gap) — out of scope for a normal send but worth an honest note. PARTIAL PAYMENTS: tfPartialPayment (0x00020000) is FORBIDDEN for XRP-to-XRP (temBAD_SEND_XRP_PARTIAL) — never set it on a native send. For crediting/receiving, always read the metadata `delivered_amount`, NOT the Amount field (partial-payment exploit). DELIVERED-AMOUNT honesty: history/confirmation must use delivered_amount. disallowIncomingXRP flag (lsfDisallowXRP) is advisory — recipient may signal they don't want XRP (verified live true on the exchange account); honor it as a warning. LastLedgerSequence: compose/sign step MUST set LastLedgerSequence (e.g. current ledger + ~4..20) so a tx that doesn't confirm expires deterministically instead of lingering. Standard Payment flags for a plain send: tfFullyCanonicalSig only (or none); do NOT set tfPartialPayment / tfLimitQuality for native XRP.
- **Tx construction (wallet-core):** wallet-core (WalletCore) supports XRP via TWCoinTypeXRP and the Ripple SigningInput protobuf (verified src/proto/Ripple.proto). Top-level fields: `fee` (int64 drops), `sequence` (uint32), `last_ledger_sequence` (uint32), `account` (string), `flags` (uint64), `private_key` (bytes), `public_key` (bytes), `source_tag` (uint64). Operation oneof: `op_payment` (OperationPayment) for sends — OperationPayment = { oneof amount_oneof { int64 amount (drops, native XRP) | CurrencyAmount currency_amount } ; string destination ; uint64 destination_tag }. CurrencyAmount = {string currency, string value, string issuer} for token/IOU sends. NO native multi-output (XRP Payment is single-destination by design — one Payment per recipient). CRITICAL GAP: the SigningInput proto exposes destination_tag and source_tag but has NO Memos / InvoiceID / DeliverMin / SendMax fields — so any send that needs Memos (or InvoiceID, paths, partial-payment fields) MUST be built via the `raw_json` field of SigningInput (full XRPL JSON tx, signed with the XRP binary codec). THEREFORE the COMPOSE screen must collect NOW, so signing works later: recipient address, amount (drops for XRP / {currency,value,issuer} for tokens), destination tag (uint32, required-checked), optional source tag (uint32), optional memo(s) (-> raw_json path), the live fee (drops), the account sequence (from account_info), and last_ledger_sequence (current_ledger + buffer). ## G9 TRON

### `tron`
- **Fee model:** TRON does NOT use a gas-price model. Fees are paid in two consumable RESOURCES (a base+resource model), with TRX burned only when the resource is insufficient. (1) BANDWIDTH (a.k.a. NET) — every transaction consumes bandwidth equal to its byte size (1 byte = 1 bandwidth). Each account gets 600 FREE bandwidth/day (chain param getFreeNetLimit/freeNetLimit=600, confirmed live). More bandwidth is obtained by staking (freezing) TRX. If free+staked bandwidth is insufficient, the deficit is paid by BURNING TRX at getTransactionFee = 1000 SUN per byte (0.001 TRX/byte, confirmed live). A typical TRX transfer is ~268 bytes → ~0.268 TRX if no free/staked bandwidth, or FREE if covered by the daily 600 + staked allowance. (2) ENERGY — consumed ONLY by smart-contract execution (TRC-20 transfers, contract calls). Energy is obtained by staking TRX; deficit is burned at getEnergyFee = 100 SUN per energy unit (0.0001 TRX/energy, confirmed live). NO free energy allowance (free tier is bandwidth-only). For a SMART-CONTRACT tx the caller sets fee_limit = the max TRX (in SUN) willing to burn for energy (cap getMaxFeeLimit = 15,000,000,000 SUN = 15,000 TRX, confirmed live). slow/normal/fast presets are NOT a thing on TRON — fees are deterministic per resource availability, not a competitive priority auction. The wallet should instead derive ESTIMATED COST by: estimated_bandwidth (tx byte size) and estimated_energy (from triggerconstantcontract), subtract the user's available free+staked resources, multiply the burned remainder by the live unit prices (1000 SUN/byte, 100 SUN/energy), and present ONE accurate fee (optionally with a small safety buffer on energy for the Dynamic Energy Model, e.g. *1.0–1.3). Extra one-off burns layered on top: account-activation fee (see otherFeatures) and memo fee (see dataMemoTag).
- **Fee fetch RPC:** All via TRON FullNode HTTP API on https://api.trongrid.io (TronGrid public; alt: https://api.tronstack.io, and self-host endpoints). (1) Live unit prices + caps: POST /wallet/getchainparameters (no body) → {"chainParameter":[{"key":"getTransactionFee","value":1000},{"key":"getEnergyFee","value":100},{"key":"getCreateAccountFee","value":100000},{"key":"getCreateNewAccountFeeInSystemContract","value":1000000},{"key":"getMemoFee","value":1000000},{"key":"getMaxFeeLimit","value":15000000000}, ...]} — ALL VALUES CONFIRMED LIVE 2026-06-15. (3) Energy estimate for a TRC-20/contract tx: POST /wallet/triggerconstantcontract {"owner_address","contract_address","function_selector":"transfer(address,uint256)","parameter":<abi-encoded>,"visible":true} → {"energy_used":3459,"energy_penalty":2666,...} (energy_used INCLUDES the dynamic penalty; confirmed live for USDT). Note: /wallet/estimateenergy is MORE accurate but the public TronGrid node returned 'this node does not support estimate energy' — so triggerconstantcontract is the reliable path. (4) Recipient activation check: POST /wallet/getaccount {"address":"T...","visible":true} → empty body {} means NOT activated (confirmed live).
- **Gas / compute / weight:** No 'gas limit' concept; instead bandwidth-bytes + energy. NATIVE TRX TRANSFER (TransferContract): bandwidth ≈ 265–270 bytes (≈268 typical, confirmed by docs); ZERO energy. Determine bytes exactly from the raw_data_hex length returned by /wallet/createtransaction (confirmed live: returns raw_data_hex you can measure). TRC-20 TOKEN TRANSFER (TriggerSmartContract transfer(address,uint256)): bandwidth ≈ 345 bytes; energy is the dominant cost — estimate via /wallet/triggerconstantcontract energy_used. CONFIRMED LIVE: USDT transfer to a recipient that ALREADY holds USDT ≈ 3,459 energy in the constant-call probe, but real on-chain consumption for USDT is materially higher due to the Dynamic Energy Model + SSTORE state — docs/real-world record: ~64,000–65,000 energy to a recipient with non-zero USDT balance, ~130,000–131,000 energy to a FRESH zero-balance recipient (SSTORE 5,000 vs 20,000 + dynamic energy_factor). PRODUCTION RULE: always call triggerconstantcontract per-tx for the live estimate, add a safety buffer (e.g. +10–30%) for the dynamic model, then fee_limit = ceil(estimated_energy * getEnergyFee) capped at getMaxFeeLimit. Defaults if estimation unavailable: native transfer fee_limit not needed (no energy); TRC-20 fallback fee_limit ≈ 100 TRX (100,000,000 SUN) for USDT to be safe, but PREFER the live estimate.
- **Units & decimals:** Base unit = SUN. 1 TRX = 1,000,000 SUN (6 decimals) — TRX itself has 6 decimals. getTransactionFee is in SUN-per-byte (1000 SUN = 0.001 TRX/byte). getEnergyFee is in SUN-per-energy (100 SUN = 0.0001 TRX/energy). fee_limit is in SUN. createAccountFee 100000 SUN = 0.1 TRX; createNewAccountFeeInSystemContract 1000000 SUN = 1 TRX; memoFee 1000000 SUN = 1 TRX; maxFeeLimit 15000000000 SUN = 15000 TRX. Display: convert burned-SUN total to TRX (divide by 1e6), then multiply by the TRX/fiat rate. TRC-20 token amounts use the TOKEN's own decimals (USDT on TRON = 6 decimals, contract TR7NHqjeKQxGTCi8q8ZY4pL8otSzgjLj6t) — do NOT confuse token decimals with TRX's 6. All money math in Decimal.
- **UTXO:** N/A (account model). TRON is an account-based chain (like Ethereum) — no UTXOs, no coin selection, no change outputs, no dust limit, no RBF. Balance is read from /wallet/getaccount .balance (SUN for TRX) and per-token via triggerconstantcontract balanceOf for TRC-20. A single transfer spends from the account balance directly; one output per TransferContract (TRON does NOT support multiple outputs in one transfer contract — each TransferContract has exactly one to_address; batch sends require multiple transactions or a contract).
- **Data / memo / tag:** MEMO supported via the transaction's raw_data.data field (the Transaction proto 'memo'/'data' field; wallet-core SigningInput.Transaction.memo, field 5). Format: arbitrary bytes/UTF-8 string (commonly used for exchange deposit identifiers). CRITICAL COST: if raw_data.data is NON-EMPTY the network burns an EXTRA 1 TRX (getMemoFee = 1,000,000 SUN, confirmed live; in force since committee proposal #80, 2022-12-16). If the memo field is empty, NO memo fee. EXCHANGE REQUIREMENT: many CEX TRON (TRX) deposit addresses REQUIRE a memo/tag to credit deposits to the right user account — the compose screen MUST expose an optional memo field for TRX sends and warn that (a) a memo costs +1 TRX and (b) some exchanges require it. Query historical memo price via /wallet/getmemofee. Note: TRC-20 token transfers carry their note inside the contract data, but the TRON-level memo (and its 1 TRX fee) still applies to the raw_data.data field independently.
- **Other features:** FULL FEATURE SET a production TRON compose screen must handle: (1) ACCOUNT ACTIVATION FEE — sending TRX/TRC-10 to a NEVER-ACTIVATED address (getaccount returns empty {}) burns 1 TRX (getCreateNewAccountFeeInSystemContract = 1,000,000 SUN) PLUS 100 bandwidth; if the sender lacks bandwidth, an extra 0.1 TRX (getCreateAccountFee = 100,000 SUN) is burned for that bandwidth. (2) BANDWIDTH MODEL — 600 free bandwidth/day per account; consumption priority: staked bandwidth → free 600 → burn TRX @1000 SUN/byte; recovers over 24h. (3) ENERGY MODEL — no free allowance; obtained by staking TRX; deficit burned @100 SUN/energy; recovers over 24h; subject to DYNAMIC ENERGY MODEL (popular contracts like USDT add energy_factor penalty, energy = base*(1+energy_factor)) — always re-estimate live. (4) FEE_LIMIT — required for contract calls (TRC-20); the per-tx burn cap; if too low → OUT_OF_ENERGY, tx fails and burned energy is NOT refunded; cap 15,000 TRX. (5) RESOURCE LEASING/STAKING (Stake 2.0: FreezeBalanceV2/UnfreezeBalanceV2/DelegateResource/UndelegateResource — present in wallet-core proto) lets users acquire bandwidth/energy without burning — relevant if the wallet later adds 'rent energy'/staking, but for send-only the wallet just measures available resources and computes the burn. (6) SEND-ALL (max) for TRX: max_send = balance - (bandwidth_burn_if_any) - (memo_fee_if_memo) - (activation_fee_if_new_recipient); for TRC-20, fees come out of the TRX balance, NOT the token, so token send-all = full token balance but the account must hold enough TRX/energy to cover the burn. (7) NO priority/replace mechanism — fees are deterministic, not a fee market. (8) TRC-10 tokens also exist (TransferAssetContract with asset_name/token_id) in addition to TRC-20.
- **Tx construction (wallet-core):** wallet-core (WalletCore) FULLY supports TRON. Transaction { int64 timestamp; int64 expiration; BlockHeader block_header; int64 fee_limit; string memo; oneof contract_oneof }. contract_oneof includes: transfer (TransferContract: owner_address, to_address, amount) for native TRX; trigger_smart_contract (TriggerSmartContract: owner_address, contract_address, call_value, bytes data, call_token_value, token_id) AND transfer_trc20_contract for TRC-20; plus transfer_asset (TRC-10), freeze_balance_v2/unfreeze_balance_v2/delegate_resource/undelegate_resource/withdraw_balance/vote_witness for staking/voting. Default expiration in wallet-core = timestamp + 10h (TRON node default is timestamp+60s — set expiration explicitly to a short window, e.g. now+60s, to avoid stale-tx). COMPOSE SCREEN MUST COLLECT NOW so signing works later: recipient address (Base58Check T...), amount (in token base units), optional memo string (and warn of +1 TRX), the selected asset (native TRX vs TRC-20 contract address+decimals), and for TRC-20 the fee_limit (computed from live energy estimate). ## G10 NEAR

### `near`
- **Fee model:** NEAR uses a deterministic GAS-UNIT model (NOT EIP-1559, NOT sat/vB). Every protocol action has a FIXED gas-unit cost; only FunctionCall actions take a user-chosen 'attached/prepaid gas'. Total fee = gas_units * gas_price (in yoctoNEAR/gas), deducted from the signer's balance. gas_price is a single per-block network value (NOT a user-set tip): floor is 100000000 yocto/gas (= 0.0001 NEAR per Tgas, where 1 Tgas = 10^12 gas). It recalculates every block: if the previous block was >50% full it rises +1%, else falls -1% down to the floor (live-confirmed value today = 100000000, the floor). Practical range cited by docs: 0.1 NEAR/Pgas to 2 NEAR/Pgas. Because gas_units are fixed and the user cannot bid a tip, there is NO slow/normal/fast SPEED preset for a plain transfer — speed does not vary with fee. Aperture should present a single computed 'Network fee' for a native transfer (deterministic) and, for function calls / FT transfers, expose the attached-gas budget (default 30 Tgas) as the only knob. A 'fast/normal' UI, if shown for consistency, should map ONLY to slightly padding the fetched gas_price (e.g. +0%/+10%) to survive a 1%/block rise between fetch and broadcast — NOT to a real priority market. NOTE: the 'pessimistic gas price' multiplier was REMOVED in protocol v78 / NEP-536, so no longer pad for that.
- **Fee fetch RPC:** PRIMARY (gas price): JSON-RPC method 'gas_price'. Request (latest block): {"jsonrpc":"2.0","id":"dontcare","method":"gas_price","params":[null]} -> {"result":{"gas_price":"100000000"}} (live-confirmed today on both rpc.mainnet.near.org and free.rpc.fastnear.com). Can also pass [<block_height>] or ["<block_hash>"] for historical (archival only). The gas_price is ALSO present in any 'block' header response (header.gas_price), so a single 'block' {finality:final} call yields BOTH the recent block_hash needed for the tx AND the gas_price. For exact per-action gas-unit costs (so the fee is computed without estimateGas): method 'EXPERIMENTAL_protocol_config' {finality:final} -> result.runtime_config.transaction_costs.{action_receipt_creation_config, action_creation_config.transfer_cost} and result.runtime_config.storage_amount_per_byte. Live-confirmed today: transfer_cost = {send_sir:115123062500, send_not_sir:115123062500, execution:115123062500}; action_receipt_creation_config = {send_sir:108059500000, send_not_sir:108059500000, execution:108059500000}; storage_amount_per_byte = 10000000000000000000.
- **Gas / compute / weight:** NATIVE NEAR transfer gas is DETERMINISTIC, NOT eth_estimateGas-style. Total = receipt-creation cost (send + execution) + Transfer-action cost (send + execution). From live protocol config: (108059500000+108059500000) + (115123062500+115123062500) = 446365125000 gas ~= 0.446 Tgas. Docs round this to ~0.45 Tgas = 0.000045 NEAR at the floor gas_price. So a native transfer fee ~= 446365125000 * 100000000 yocto = 4.46e19 yocto = 0.0000446 NEAR. Aperture can hardcode/derive from protocol config rather than calling an estimator (NEAR has no eth_estimateGas; there IS a 'EXPERIMENTAL_tx_status'/dry-run path but it is unnecessary for a plain transfer). FT/FUNCTION-CALL transfer (NEP-141 ft_transfer): use the standard ATTACHED/PREPAID gas budget of 30 Tgas = 30000000000000 gas (near-api-js default; FT reference impl requires >=30 Tgas). Unused attached gas is REFUNDED, minus a refund fee of at least 1 Tgas or 5% of the unspent gas. Per-tx hard cap = 300 Tgas (300000000000000). Max function-call gas burnt limit (wasm) live-confirmed = 1000000000000000. Defaults: native transfer = derive from protocol config (~0.45 Tgas, fully refunded if over); token (NEP-141) transfer = 30 Tgas attached.
- **Units & decimals:** Base unit = yoctoNEAR. 1 NEAR = 10^24 yoctoNEAR (decimals = 24). gas_price is yoctoNEAR PER gas-unit; gas-unit subunit Tgas = 10^12 gas, Pgas = 10^15 gas. Fee in NEAR = (gas_units * gas_price) / 10^24. All on-wire amounts (account 'amount', 'locked', deposit, gas_price, storage_amount_per_byte) are decimal STRINGS of yoctoNEAR and MUST be handled as 128-bit integers / Swift Decimal (never Double — 10^24 overflows Double precision). Display: divide yocto by 10^24 for NEAR; multiply by fiat price for fiat. FT amounts use the token's OWN decimals from ft_metadata (live-confirmed: usdt.tether-token.near symbol=USDt decimals=6) — NOT 24.
- **UTXO:** N/A (account model). NEAR is an account-based chain — no UTXOs, no coin selection, no dust limit, no change outputs, no RBF. Sending is a balance debit on the signer account. Replacement: a stuck tx is resubmitted only by signing a NEW tx with the next nonce (there is no fee-bump/RBF concept).
- **Data / memo / tag:** Native NEAR transfers (Transfer action) have NO memo/OP_RETURN/tag field — the Transfer action carries ONLY a 'deposit' (amount). Therefore exchanges that need to credit a shared NEAR deposit address use a DEDICATED named subaccount per user (not a memo). HOWEVER: NEP-141 fungible-token transfers DO support a memo: ft_transfer(receiver_id, amount, memo?: string) — an OPTIONAL free-form string (no fixed max in the standard; bounded by tx size / 300 Tgas budget). Some FT-based exchange flows use this memo. Aperture should: (a) for native NEAR — no memo UI; if a destination requires routing, it is via the receiver account_id itself; (b) for NEP-141 token sends — expose an optional 'memo' text field passed into the ft_transfer args. Also note a general 'function-call args' JSON can carry arbitrary data on any FunctionCall action.
- **Other features:** COMPOSE-CRITICAL items, do not omit any: (1) ACCOUNT MODEL — receiver can be a named account (e.g. alice.near, max 64 chars, lowercase a-z0-9-_ and dots) OR a 64-hex-char implicit account; validate accordingly. Sending native NEAR to an account that does NOT yet exist will CREATE an implicit account only if the receiver is a 64-hex implicit id; sending to a non-existent NAMED account fails. (2) STORAGE-STAKING RESERVE / MIN BALANCE — every account must keep enough NEAR to cover its state: rate = storage_amount_per_byte = 10^19 yocto/byte = 0.00001 NEAR/byte (live-confirmed) = 100KB per 1 NEAR. A basic account with 1 full-access key needs a MINIMUM ~0.00182 NEAR locked (varies with storage_usage from view_account, e.g. storage_usage * 10^19 yocto). SEND-ALL / MAX must = amount - locked - (storage_usage * storage_amount_per_byte) - estimated_fee; NEVER allow draining below the storage reserve or the tx fails / account becomes unusable. 'locked' field from view_account is staked-for-validation and is also unavailable. (3) AVAILABLE BALANCE = amount(total yocto) - locked - storage_reserve. (4) NEP-141 TOKEN SEND PRECONDITIONS — receiver must be REGISTERED with the token contract (NEP-145 storage). Check storage_balance_of(account_id); if null, prepend a storage_deposit call. Registration min live-confirmed via storage_balance_bounds on a real FT = 1250000000000000000000 yocto = 0.00125 NEAR (min==max for fixed-bound tokens). ft_transfer itself REQUIRES exactly 1 yoctoNEAR attached deposit (security: blocks fn-call-key phishing). (5) ACCESS KEYS — txs are signed by a FullAccess ed25519 key; the tx NONCE must be (current access-key nonce + 1), fetched per-key. Wallet may also hold FunctionCall-restricted keys (allowance + receiver + method whitelist) — relevant if Aperture later supports dApp keys. (6) NONCE & BLOCK_HASH must be fetched fresh just before signing (block_hash must be from the last ~24h or the tx is rejected as expired) — fits Rule #27 just-in-time sync. (7) TIP — none (no priority tip; gas_price is network-set). (8) NO contract gas estimator needed for native transfer (deterministic).
- **Tx construction (wallet-core):** wallet-core (WalletCore) FULLY supports NEAR. Use TW.NEAR.Proto.SigningInput with: signer_id (string), nonce (uint64 = access-key nonce + 1), receiver_id (string), block_hash (bytes = base58-DECODED recent block hash), actions (repeated Action), private_key (32-byte ed25519), public_key (32-byte ed25519). NATIVE multi-action IS supported (actions is repeated) — one tx can batch e.g. [storage_deposit FunctionCall, ft_transfer FunctionCall]. ACTION TYPES available in proto: CreateAccount, DeployContract, FunctionCall{method_name, args(bytes JSON), gas(uint64), deposit(16-byte u128)}, Transfer{deposit(16-byte u128)}, Stake{stake, public_key}, AddKey{public_key, AccessKey{nonce, FullAccess|FunctionCall{allowance,receiver_id,method_names}}}, DeleteKey, DeleteAccount{beneficiary_id}, plus a convenience TokenTransfer{token_amount(decimal string), receiver_id, gas(uint64), deposit(16-byte u128)} that wallet-core internally serializes into an ft_transfer FunctionCall. CRITICAL BYTE-ORDER FACT (verified in src/NEAR/Serialization.cpp writeU128): the 'deposit'/'allowance'/'stake' bytes are copied VERBATIM (no byte-swap) and NEAR Borsh serializes u128 as LITTLE-ENDIAN — so the compose/sign layer MUST pass the amount as a 16-byte LITTLE-ENDIAN array (the proto comment that says 'big endian' is WRONG; the wallet-core unit test sets deposit[0]=1 for value 1, confirming LE). For a NATIVE send: one Transfer action, deposit = amount_in_yocto as 16-byte LE. For a TOKEN send: TokenTransfer with token_amount=decimal string in the token's own decimals, gas=30000000000000 (30 Tgas), deposit = 1 yoctoNEAR as 16-byte LE (=[1,0,0,...]). COMPOSE SCREEN MUST COLLECT NOW (so signing works later): receiver account_id, amount (+ token contract & its decimals for FT), an optional FT memo, AND it must trigger a just-in-time fetch of {access-key nonce, recent block_hash, gas_price} at sign time. SigningOutput returns signed_transaction (bytes, base64 for send_tx) + hash. BROADCAST: prefer 'send_tx' {params:{signed_tx_base64, wait_until:'EXECUTED_OPTIMISTIC'}} (new canonical); legacy 'broadcast_tx_commit' / 'broadcast_tx_async' (positional [base64]) still work. Status: 'tx' / 'EXPERIMENTAL_tx_status' {tx_hash, sender_account_id}.

## G11 Polkadot

### `polkadot`
- **Fee model:** WEIGHT-BASED, NOT gas, NOT sat/vB. The fee is an 'inclusion fee' computed by the runtime: inclusion_fee = base_fee + length_fee + (targeted_fee_adjustment * weight_fee), and final_fee = inclusion_fee + tip. (1) base_fee = ExtrinsicBaseWeight converted via WeightToFee (covers signature verification / inclusion overhead; ~0.001 DOT order of magnitude per polkadot-developers knowledgebase). (2) length_fee = encoded_extrinsic_byte_length * TransactionByteFee (per-byte). (3) weight_fee = the dispatch's weight (from #[pallet::weight] + ExtrinsicBaseWeight) run through the runtime WeightToFee Convert<Weight,Balance> struct. (4) targeted_fee_adjustment (a.k.a. NextFeeMultiplier / FeeMultiplier) is a DYNAMIC congestion multiplier: if the previous block exceeded the target block-fullness, the multiplier rises slightly, so identical transfers cost slightly more under congestion and the multiplier decays back when blocks are empty. This is why the fee MUST be queried live per-tx and not hardcoded. partial_fee returned by the RPC is precisely the inclusion_fee EXCLUDING tip (authoritative: pallet_transaction_payment_rpc_runtime_api::RuntimeDispatchInfo — 'the inclusion fee of this dispatch. This does not include a tip or anything else that depends on the signature (i.e. depends on a SignedExtension)'). Slow/normal/fast PRESETS are NOT a fee-rate slider like EVM/BTC — Polkadot has no fee market the sender bids into for the base fee. The only sender-controlled lever is the optional TIP (priority). Recommended compose UX: show partial_fee as the (non-negotiable) network fee, and offer tip presets — Normal = tip 0; Fast/Priority = a small positive tip (e.g. a fixed small DOT amount or a percentage of partial_fee). Unused weight is refunded post-dispatch for transfers that come in under their worst-case weight, so the actually-charged fee can be <= the estimate.
- **Fee fetch RPC:** TWO live RPC paths, both confirmed working on rpc.polkadot.io AND polkadot-rpc.publicnode.com on 2026-06-15. (A) DEPRECATED-but-still-live JSON-RPC: payment_queryInfo — request {"jsonrpc":"2.0","id":1,"method":"payment_queryInfo","params":["<0x SCALE-encoded SIGNED extrinsic hex>", <optional blockHash>]} → returns {weight:{refTime,proofSize}, class, partialFee:"<plancks as decimal string>"}. Companion payment_queryFeeDetails → {inclusionFee:{baseFee, lenFee, adjustedWeightFee}} (each in plancks) for the itemized breakdown. (B) MODERN runtime API via state_call: {"method":"state_call","params":["TransactionPaymentApi_query_info","<0x: SCALE(uxt) ++ SCALE(len u32)>"]} and "TransactionPaymentApi_query_fee_details" — returns the SCALE-encoded RuntimeDispatchInfo / FeeDetails to decode. BOTH require a fully SCALE-encoded extrinsic (a dummy-signed one is fine — apply a throwaway 64-byte signature; partialFee is signature-independent). Live-verified error shapes: empty/garbage params return 'Could not decode OpaqueExtrinsic.0 / Not enough data to fill buffer' (proves method reachable; needs real extrinsic). RECOMMENDED for Aperture: build the dummy-signed extrinsic with wallet-core, then call payment_queryInfo (simplest, returns JSON with partialFee directly) — no client-side SCALE decode needed. Supporting RPCs needed to BUILD the extrinsic for that query: state_getRuntimeVersion (specVersion+transactionVersion — LIVE = specVersion 2003000, transactionVersion 26), chain_getBlockHash with param [0] (genesis = 0x91b171bb158e2d3848fa23a9f1c25182fb8e20313b2c1eb49219da7a70ce90c3), chain_getBlockHash [] / chain_getFinalizedHead (mortal-era checkpoint hash), system_accountNextIndex [ss58Addr] (nonce — LIVE-verified returns integer, e.g. 0).
- **Gas / compute / weight:** There is NO gas limit and NO eth_estimateGas equivalent the sender sets. The dispatch WEIGHT is fixed by the runtime per call (benchmarked into #[pallet::weight]) and is returned inside the payment_queryInfo response's `weight` field ({refTime, proofSize} — Weight v2). The sender does not estimate or cap weight; the runtime knows it. For a NATIVE DOT transfer the call is balances.transferKeepAlive (default) or balances.transferAllowDeath; its weight is whatever the live runtime benchmark says (read from payment_queryInfo, do not hardcode). For a TOKEN transfer (DOT has no relay-chain tokens; assets live on Asset Hub parachain — see otherFeatures): assets.transfer / assets.transferKeepAlive on Asset Hub, weight again returned by that parachain's payment_queryInfo. Practical default for the compose screen: do NOT show a 'gas limit' field; show only the live partialFee + optional tip. The only thing the compose screen must compute is the encoded extrinsic length (the SCALE byte length) which feeds the len arg of the runtime API / is implicit in payment_queryInfo's extrinsic.
- **Units & decimals:** Native unit: DOT. Smallest unit: PLANCK. 1 DOT = 10^10 plancks (tokenDecimals = 10 — LIVE-confirmed via system_properties: {ss58Format:0, tokenDecimals:10, tokenSymbol:'DOT'}; matches wallet-core registry decimals=10). ALL on-chain values (free balance from System.Account, partialFee, inclusionFee components, tip, transfer value) are integers in plancks. Conversion for display: dotAmount = planckAmount / 10^10 (use Swift Decimal, never Double — a u128 plancks balance overflows Double precision). Fiat: dotAmount * dotUsdPrice. NOTE Kusama (not in this group) is 12 decimals / KSM — do not reuse this 10^10 factor for other Substrate chains.
- **UTXO:** N/A (account model). Polkadot uses the Substrate account/nonce model (AccountId32 + System.Account nonce), not UTXOs. There is no UTXO set to fetch, no coin selection, no change output, no RBF. Balance is read from the System.Account storage map (free/reserved/frozen/flags u128 each) — Aperture's existing PolkadotChainAdapter already does this via state_getStorage with key twox128('System')++twox128('Account')++blake2_128(accountId)++accountId, decoding data.free at offset 16. 'Dust' analogue is the EXISTENTIAL DEPOSIT (see otherFeatures) — an account whose free balance would drop below ED is REAPED and its remaining funds DESTROYED; this is the dust-equivalent guard the compose screen must enforce.
- **Data / memo / tag:** No native protocol-level memo/tag/OP_RETURN field on a balances transfer. The relay-chain balances.transferKeepAlive / transferAllowDeath extrinsics take ONLY (dest: MultiAddress, value: Compact<Balance>) — there is no destination tag and exchanges that need to disambiguate deposits use a UNIQUE PER-USER ADDRESS, never a memo. IMPORTANT wallet-core convenience: TWPolkadot's Balance.Transfer proto HAS a `memo` field (string, field 3) — but wallet-core implements this by wrapping the transfer in a utility.batchAll with a system.remarkWithEvent('memo') call, NOT a native transfer field. Most exchanges do NOT parse such remarks; treat memo as UNSUPPORTED for exchange deposits and surface a per-user address instead. If an exchange ever documents requiring a remark, Aperture can set transfer.memo and wallet-core builds the batch. So for the compose screen: do NOT show a memo/tag field for DOT by default (collect a unique deposit address); the underlying capability exists if ever needed.
- **Other features:** EXISTENTIAL DEPOSIT (ED) — CRITICAL, must be enforced in compose: DOT ED = 0.01 DOT = 100,000,000 plancks (official Polkadot Support; the historical 1 DOT was lowered). An account whose FREE balance falls below ED is reaped and ALL remaining free funds are DESTROYED. Two protections the compose screen must implement: (1) KEEP-ALIVE vs ALLOW-DEATH transfer choice — DEFAULT to balances.transferKeepAlive (pallet index 5; the runtime rejects the tx rather than reaping the SENDER if it would drop sender below ED). Offer balances.transferAllowDeath only for an explicit 'send and close account' intent. (2) RECIPIENT ED guard — sending an amount that leaves the RECIPIENT (if new/empty) below ED FAILS; warn if amount < ED to a fresh account. SPENDABLE/TRANSFERABLE balance (the real max the user can send) = free - max(frozen - reserved, ED) (official wiki spendable formula). Compose must base 'available' and 'Max' on spendable, NOT free. SEND-ALL / MAX: use balances.transferAll(dest, keep_alive: bool) which moves the entire transferable balance in one extrinsic and auto-accounts for the fee — set keep_alive=true to leave ED behind (default), keep_alive=false to also sweep ED and reap own account. This is the correct 'Max' button implementation (avoids the user manually subtracting fee+ED and failing). TIP: optional priority fee, Compact<Balance> in plancks, added on top of partial_fee; expose as the only fee lever (presets: none / small). MORTAL vs IMMORTAL ERA: default to a MORTAL extrinsic (era block_number = current finalized block, period = 64 blocks ≈ ~6.4 min validity) so a stuck tx auto-expires and can't be replayed after reaping; immortal (era 0x00) is replay-risky and discouraged. NONCE: from system_accountNextIndex; it does NOT account for pending tx — Aperture must track/increment locally if sending multiple. RESERVED balance (staking/deposits) is NOT spendable and adds a provider ref protecting the account from reaping. FROZEN balance (vesting/governance locks) also reduces spendable. Asset Hub tokens (USDC asset id 1337 etc. per existing PolkadotAssetRegistry) live on a SEPARATE parachain endpoint with its OWN assets pallet + its own ED per asset + the option to pay fees in the asset itself (ChargeAssetTxPayment / wallet-core charge_native_as_asset_tx_payment & AssetTransfer.fee_asset_id) — out of relay-chain scope but the compose layer must route asset-id transfers to the Asset Hub adapter.
- **Tx construction (wallet-core):** wallet-core (WalletCore) FULLY supports Polkadot. Registry: id 'polkadot', coinId/slip44 354, decimals 10, curve ed25519 (NOT sr25519 — wallet-core uses ed25519 for DOT), publicKeyType ed25519, ss58Prefix 0, derivationPath m/44'/354'/0'/0'/0'. SigningInput = TWPolkadotProto.SigningInput (src/proto/Polkadot.proto), fields the compose/sign step MUST collect: block_hash (bytes, field 1 — mortal-era checkpoint hash from chain_getBlockHash), genesis_hash (bytes, field 2 — chain_getBlockHash[0] = 0x91b171bb...c90c3, can be a constant for relay chain), nonce (uint64, field 3 — system_accountNextIndex), spec_version (uint32, field 4 — state_getRuntimeVersion LIVE 2003000, MUST fetch live), transaction_version (uint32, field 5 — state_getRuntimeVersion LIVE 26, MUST fetch live; wallet-core test fixtures show stale 5 — do NOT hardcode), tip (bytes, field 6 — raw value bytes, empty=0), era (Era msg field 7: block_number uint64 + period uint64; for mortal set block_number=finalized block height, period=64), private_key (bytes, field 8 — supplied at sign time only), network (uint32, field 9 — set to ss58Prefix = 0 for Polkadot), multi_address (bool, field 10 — set TRUE for modern MultiAddress runtimes; current Polkadot runtime uses MultiAddress so dest is wrapped as {Id: AccountId32}), charge_native_as_asset_tx_payment (bool, field 13 — Asset Hub only). message_oneof: balance_call (field 11) → Balance.transfer { to_address (string SS58, field 1), value (bytes, field 2 — the raw u128 amount in plancks; wallet-core SCALE-compact-encodes it internally), memo (string, field 3 — optional, builds utility.batch+system.remark), call_indices (field 4 — OPTIONAL override of {pallet_index, method_index}; if omitted wallet-core uses its built-in Polkadot defaults). Balances pallet = index 5; transferAllowDeath method index 0 (call 0x0500), transferKeepAlive method index 3 (call 0x0503), transferAll method index 4 (call 0x0504) — provide via call_indices if a runtime upgrade changes defaults; otherwise rely on wallet-core defaults. Signing curve ed25519 means the signed payload >256 bytes is blake2b-256 hashed before signing (wallet-core handles). Compose screen must therefore persist for the sign step: recipient SS58, amount in plancks, chosen transfer kind (keepAlive default / allowDeath / transferAll+keep_alive flag), tip, and the live-fetched specVersion/transactionVersion/genesisHash/blockHash/nonce/era. AnyAddress validation: TWAnyAddress with TWCoinTypePolkadot (ss58Prefix 0) — reject Kusama(2)/generic(42) prefixes (Aperture's SS58.decodeAccountId already enforces prefix 0).

## G12 Kava / Cosmos

### `kava`
- **Fee model:** Cosmos SDK base+priority-less FIXED fee model (NO EIP-1559, NO base-fee burning on the Cosmos side). The total fee a tx pays = gas_limit (uint64, called `gas` in the Fee message) × gas_price (a decimal price per gas unit in a denom). The user only controls two inputs: (1) the GAS LIMIT (how many gas units the tx is allowed to consume) and (2) the GAS PRICE (price per unit, e.g. 0.05ukava). The fee carried in the tx is the ABSOLUTE amount = ceil(gas_limit × gas_price) in ukava, placed in `auth_info.fee.amount` as a Coin {denom:"ukava", amount:"<int>"}. Validators reject txs whose offered gas_price is below their node's `minimum-gas-prices`. LIVE-VERIFIED node config (docs.kava.io node-config FAQ) sets minimum-gas-prices = "0.001ukava;1000000000akava" — i.e. the floor a node will accept is 0.001 ukava/gas (and 1e9 akava/gas for EVM-side txs). The chain-registry (cosmos/chain-registry kava/chain.json) advertises wallet fee tiers in ukava/gas: LOW=0.05, AVERAGE=0.1, HIGH=0.25 (no fixed_min_gas_price field present). PRESET DERIVATION for the compose screen: slow=0.05ukava/gas, normal=0.1ukava/gas, fast=0.25ukava/gas (chain-registry tiers, which sit well above the 0.001 node floor). Kava CLI examples use 0.01ukava/gas and a fixed gas of 650000 (docs.kava.io quick-guide) — that 0.01 also clears the 0.001 floor but is below the registry low tier; safest default for the wallet is the registry AVERAGE (0.1) for 'normal'. Final fee = gas_limit × selected_tier, rounded UP to an integer ukava. Note Kava is a dual-VM chain: the Cosmos side fees in ukava (6dp), the co-resident EVM side fees in akava (18dp) via eth_gasPrice — but a Cosmos bank send (the compose screen's transfer path) uses the ukava model described here.
- **Fee fetch RPC:** Live fee inputs are fetched from the Cosmos REST (LCD) / gRPC, NOT from a single 'gas price oracle'. Three calls:
(1) GAS PRICE preset: static chain-registry tiers (low 0.05 / avg 0.1 / high 0.25 ukava) — fetch/refresh from https://raw.githubusercontent.com/cosmos/chain-registry/master/kava/chain.json (fees.fee_tokens[0]), or hardcode-with-periodic-refresh. There is no per-block dynamic gas-price RPC on the Cosmos side (no feemarket module active for bank sends).
(2) GAS LIMIT estimate via SIMULATE — POST https://<lcd>/cosmos/tx/v1beta1/simulate  Content-Type: application/json  body: {"tx_bytes":"<base64 of a fully-built (can be signed-with-dummy or with skip_simulate flag) TxRaw>"}. Response: {"gas_info":{"gas_wanted":"...","gas_used":"<N>"}, "result":{...}}. Take gas_used, multiply by gas_adjustment (1.3–1.5) to get gas_limit. LIVE-VERIFIED reachable: empty body returns {"code":3,"message":"empty txBytes is not allowed"} (endpoint exists & validates). gRPC equivalent: cosmos.tx.v1beta1.Service/Simulate with SimulateRequest{tx_bytes}.
(3) account_number + sequence (needed to build the tx to simulate AND to sign) via GET https://<lcd>/cosmos/auth/v1beta1/accounts/<kava-address> → account.account_number, account.sequence (LIVE-VERIFIED: returned account_number "13751", sequence "3175" for a real address; vesting accounts nest these under base_vesting_account.base_account). LCD endpoint used in tests: https://kava-rest.publicnode.com (chain-registry lists 9 REST providers incl. api.data.kava.io, kava-api.polkachu.com).
- **Gas / compute / weight:** Two paths, both standard Cosmos. (A) STATIC DEFAULT (no network round-trip): a single bank MsgSend of native ukava costs ~80,000–120,000 gas in practice; Kava CLI hardcodes --gas 650000 for general txs (docs.kava.io quick-guide) as a safe ceiling. Recommended compose-screen default gas_limit for a native ukava transfer: 200,000 (covers MsgSend + signature verification with margin). For an IBC/CW20-style token transfer it would be higher (~250,000–400,000), but on the Cosmos side most 'tokens' Kava holds are native denoms (ukava, swp, usdx, hard, ibc/... — LIVE-VERIFIED in a real balance) sent via the SAME MsgSend, so the same ~200,000 default applies; a wallet-side token transfer is just a MsgSend with a different denom, not a contract call. (B) ACCURATE: call /cosmos/tx/v1beta1/simulate, read gas_info.gas_used, then gas_limit = ceil(gas_used × gas_adjustment) where gas_adjustment defaults to 1.3 (cosmos CLI --gas auto uses 1.0 then you multiply; standard wallets use 1.3–1.5 because gas_used at simulate time slightly understates real execution). Per-transfer defaults to collect on the compose screen: gas_limit (default 200000, editable in 'advanced'), gas_adjustment (1.3 if using simulate). NOTE: the EVM side (akava) would use eth_estimateGas / 21000, but that is a separate path not used by the Cosmos bank-send compose flow.
- **Units & decimals:** Cosmos side: base unit = ukava (micro-KAVA), 6 decimals. 1 KAVA = 1,000,000 ukava (10^6). All on-chain amounts (send amount, fee amount, balances) are integer strings in ukava. Display: native amount = ukava / 1e6; fiat = (ukava / 1e6) × KAVA_USD. Gas PRICE is a decimal in ukava-per-gas (e.g. 0.1ukava); fee amount = ceil(gas_limit × gas_price) ukava. EVM side (separate VM, NOT the bank-send path): base unit = akava, 18 decimals, 1 KAVA = 10^18 akava; conversion akava↔ukava multiplier = 10^12 (1 ukava = 1,000,000,000,000 akava) — LIVE/doc-VERIFIED (Kava EvmBankKeeper, x/evmutil). The compose screen for a Cosmos transfer formats everything in ukava (6dp); only the EVM-transfer path (if added) uses akava (18dp). Fee denom for Cosmos bank send = ukava ALWAYS (bond_denom is ukava — LIVE-VERIFIED staking params).
- **UTXO:** N/A (account model). Kava Cosmos is account-based: each address has an auth account with account_number + sequence (nonce). No UTXO set, no coin selection, no change outputs, no dust limit, no RBF. The 'spendable' set is the address's bank balances map (denom→amount). Fetch via GET /cosmos/bank/v1beta1/balances/<addr> (all denoms, paginated — LIVE-VERIFIED returns ukava/swp/usdx/ibc-denoms) or /cosmos/bank/v1beta1/balances/<addr>/by_denom?denom=ukava for a single denom (LIVE-VERIFIED returns {denom:ukava, amount:N}). Replace-by-fee analog: resubmit with the same sequence (a tx with a stale/duplicate sequence is rejected; to 'replace' you must use the SAME sequence before the first lands, which is not a user-facing wallet feature here). Sequence MUST be fetched fresh before each send (LIVE account had sequence 3175).
- **Data / memo / tag:** MEMO field (the Cosmos universal 'tag'). Lives in TxBody.memo (string). LIVE-VERIFIED chain limit via /cosmos/auth/v1beta1/params: max_memo_characters = 512 (so the compose screen must cap the memo input at 512 UTF-8 characters). Format: free-form UTF-8 string. EXCHANGE-REQUIRED: YES for many CEX deposits — Kava (like most Cosmos chains) commonly uses a shared deposit address + per-user memo to credit deposits; a missing/wrong memo on a CEX deposit loses funds, so the compose screen MUST expose an optional 'Memo' input, warn prominently when sending to an exchange, and pass it into TxBody.memo. There is NO separate 'destination tag' concept — the memo IS the tag on Cosmos. (No OP_RETURN — that is Bitcoin-only.) Memo also costs gas: tx_size_cost_per_byte = 10 gas/byte (LIVE-VERIFIED auth params), so a long memo raises gas_used — re-simulate after the user types a memo if using accurate gas.
- **Other features:** GAS PRICE & FLOOR: validators enforce per-node minimum-gas-prices = 0.001ukava (and 1e9 akava for EVM) — offered gas_price must be ≥ floor; wallet presets (0.05/0.1/0.25) all clear it. FEE GRANTER (feegrant module, supported): AuthInfo.fee has two optional address fields — `payer` and `granter`. If `granter` is set, the fee is paid from the granter's account via an existing fee-grant allowance (MsgGrantAllowance) instead of the sender's balance; if `payer` is set, that account (which must be a signer) pays. The compose screen's 'advanced' can expose a fee-granter address for sponsored sends. SEND-ALL / MAX: because fee is deducted from the SAME ukava balance, max-send = balance_ukava − fee_ukava (= gas_limit × gas_price); always subtract the computed fee before setting the amount, and re-derive when gas/price changes (and after memo edits since memo adds gas). MULTI-DENOM: one address holds many denoms (ukava, usdx, swp, hard, ibc/...) — token transfer = MsgSend with that denom; fee STILL paid in ukava, so the user must hold some ukava to send a non-ukava token (compose screen must check ukava fee balance separately). TIMEOUT_HEIGHT (optional anti-stuck): TxBody.timeout_height can be set to (latest_height + ~50) so a tx auto-expires if not included (latest height LIVE-VERIFIED 21184993). MULTI-OUTPUT: bank supports MsgMultiSend (one tx, many recipients) — not needed for a single transfer but available. NO TIP field on modern Cosmos (deprecated). VESTING accounts (LIVE-VERIFIED a PeriodicVestingAccount exists) have locked vs spendable ukava — use /cosmos/bank/v1beta1/spendable_balances/<addr> to show only spendable for max-send when the account is a vesting account. ACCOUNT ACTIVATION: no minimum/existential deposit on Cosmos — a brand-new recipient address needs no activation and no reserve (unlike XRP/Stellar/Polkadot).
- **Tx construction (wallet-core):** WalletCore: Kava is a registered Cosmos coin (registry.json: id=kava, coinId/slip44=459, decimals=6, blockchain=Cosmos, curve=secp256k1, publicKeyType=secp256k1, addressHasher=sha256ripemd, hrp/bech32=kava, chainId=kava_2222-10, derivationPath m/44'/459'/0'/0/0) — VERIFIED. Use TWCoinType .kava (or CoinType 459) and the Cosmos.proto SigningInput. SigningInput fields to populate: signing_mode = Protobuf (Stargate/Direct sign — value 1; NOT the legacy JSON/Amino mode 0), account_number (uint64 from auth query), chain_id = "kava_2222-10", sequence (uint64 from auth query), fee = Fee{ gas: <gas_limit uint64>, amounts:[Amount{denom:"ukava", amount:"<fee int>"}] }, memo (string, ≤512), messages:[ Message with send_coins_message = MsgSend{ from_address, to_address, amounts:[Amount{denom:"ukava", amount:"<send int>"}] } ], private_key (32 bytes), and optionally public_key, mode (BroadcastMode), timeout_height. Amount is Coin{string denom, string amount} (string amounts — VERIFIED, avoids int64 overflow). SigningOutput returns `serialized` (base64 TxRaw, ready to broadcast) and `error`. THINGS THE COMPOSE SCREEN MUST COLLECT NOW so signing works later: recipient kava-bech32 address (validate hrp=kava + bech32 checksum — invalid checksum is rejected, LIVE-VERIFIED error), send amount (ukava int), denom (ukava or other held denom), memo (≤512), chosen gas_price tier + gas_limit, AND it must fetch account_number + sequence at SIGN time (sequence is consumed per tx). BROADCAST after signing: POST /cosmos/tx/v1beta1/txs with {"tx_bytes":"<serialized>", "mode":"BROADCAST_MODE_SYNC"} (sync = wait for CheckTx). Build flow: build unsigned tx with dummy sig → simulate for gas → set fee → sign with WalletCore → broadcast.
